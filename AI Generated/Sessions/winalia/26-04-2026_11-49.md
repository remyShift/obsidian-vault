- 🔴 `middleware-security.ts` exporte `securityMiddleware` jamais appelé depuis `middleware.ts` → CSP, HSTS, X-Frame-Options absents en prod
- 🔴 Routes `/api/test` (sans auth) et `/api/test-stats` exposées en prod
- 🔴 Rate limiter en mémoire (`Map<string, RateLimitEntry>`) alors que `@upstash/ratelimit` est installé — bypass trivial sur Vercel multi-instance

## Découvertes
- Le projet a été poussé en 1 seul commit "Initial commit — Winalia platform" sans historique (potentiellement migré depuis un autre repo ou copié-collé en bloc)
- La logique métier est massivement en PL/pgSQL (RPC `complete_match`, `record_ranked_match_result`, `add_to_wallet`, `subtract_from_wallet`) → modèle anémique côté TS, logique non testable en TS
- `@supabase/auth-helpers-nextjs` au root + dans apps/web alors que `@supabase/ssr` (le remplaçant officiel) est aussi présent → cohabitation deprecated/moderne
- `apps/web/package.json` force `next dev --webpack` au lieu de Turbopack par défaut (Next 16) → dev plus lent
- Présence de scripts ad-hoc à la racine de apps/web : `fix-all-params.js`, `fix-params-await.sh`, `fix-routes.sh`, `check-env.js` + 4 HTML de debug (`debug-test.html`, `test-local.html`, `test-simple.html`, `test-supabase.html`)
- Le README mentionne `PRODUCTION_CHECKLIST.md`, `SUPABASE_HEADER_REQUIREMENTS.md`, dossier `supabase/` qui n'existent pas
- 22 bounded contexts naturels détectés (wallet, auth, ranked, winamatch, shop, teams, profile, watch, premium, battle-pass, affiliates, referrals, inventory, notifications, settings, dashboard, rankings, stats, users, user, matches) sans aucune isolation
- Bon point : tsconfig.base strict, validation Zod ~85%, hCaptcha présent, KYC workflow présent, Radix UI partout (a11y gratuite), webhooks signés (Viva HMAC SHA256, Xsolla auth header)
- Bon point : routes RGPD `delete-account` et `data-export` déjà codées
- Sur une plateforme à argent réel, un simple service layer (option économique) ne suffit pas : la logique métier reste invisible, les tests deviennent des tests d'intégration Supabase, et il n'y a pas de réversibilité sur les providers
- Strangler Fig pattern (Martin Fowler) = bon compromis pour migrer 142 routes sans big bang : on construit le nouveau à côté de l'ancien et on bascule par feature flag
- Money.ts comme premier livrable galon : 80 lignes de Value Object + 10 tests Vitest = preuve d'approche peu coûteuse avant d'engager 12 semaines
- L'isolation domaine se vérifie objectivement : `grep -r "supabase.from" packages/domain packages/application` doit retourner 0 résultat ; tests domaine doivent tourner en < 2 sec sans Docker

## Décisions prises
- Approche audit en 2 livrables distincts : un technique détaillé (fichier:ligne, extraits code, références notes Obsidian) et un produit pour décideurs (vulgarisé, focus business/risque/coût)
- Verdicts tranchés par section (🔴 Critique / 🟠 À corriger / 🟡 OK) avec hiérarchisation des findings
- Plan de remédiation découpé en 4 sprints techniques (Sprint 1 = bloquants sécu en 1 sem, Sprint 2 = CI/CD/observabilité, Sprint 3 = tests + archi, Sprint 4+ = conformité)
- Plan vulgarisé pour CEO en 6 semaines (S1-2 stop the bleeding, S3-4 outillage, S5-6 tests/qualité)
- Recommandation explicite de gel des features pendant la remédiation
- Recommandation d'avis juridique gambling avant tout investissement marketing (budget 3-8 k€)
- Adopter Hexagonal Architecture (Ports & Adapters) + DDD tactique sur Winalia (pas un simple service layer, vu la criticité argent réel)
- Structure monorepo cible : `packages/{domain, application, infrastructure, contracts, ui, tsconfig, eslint-config}` + nouvelle app `apps/workers/` pour cron/dispatch outbox
- Règle de dépendance Clean Architecture : `domain` ne dépend de rien, `application` dépend de `domain`+ports, `infrastructure` implémente les ports, `apps` câblent
- 8 patterns financiers obligatoires : Idempotency keys (Upstash, déjà installé), Outbox Pattern, Audit log immuable signé (hash chaîné), Repository pattern strict, Result type (pas de throw dans le domaine), CQRS léger (writes/reads séparés), Saga pour flux multi-étapes, Event sourcing wallet (recommandé fortement)
- Migration progressive Strangler Fig en 8 étapes, ~12-13 semaines, en parallèle des features non-critiques
- Wallet en priorité absolue (Sprint 4, 3 sem) : c'est le plus critique et c'est le terrain de validation de l'approche
- Feature flag `WALLET_USE_NEW_DOMAIN` pour bascule progressive avec fallback Sentry-driven
- Périmètre négatif explicite : pas de microservices, pas d'event sourcing partout (uniquement wallet, éventuellement ranked), pas de quitter Supabase ni Next.js, pas de réécrire la couche UI (chantier composants > 400L = parallèle)
- ESLint `no-restricted-imports` pour interdire `@supabase/supabase-js` et `next/*` dans `packages/domain/` et `packages/application/`
- Standardiser sur Vitest (déjà choisi côté web), supprimer Jest de admin
- Nommage tests `should_X_when_Y` ou Given/When/Then (réf `Anatomie d'un test.md`)
- Pyramide tests cible 70/20/10 (unit/intégration/E2E, réf `Tests.md`)
- Premier pas concret cette semaine : `Money.ts` Value Object (80 L max) + 10 tests Vitest, en PR pour validation d'approche avant tout investissement long
- L'audit produit reste vierge de mention refonte (séparation stricte demandée)
