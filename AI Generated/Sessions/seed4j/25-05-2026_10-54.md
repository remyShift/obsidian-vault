## Découvertes
- Seed4J utilise déjà un module `typescript/core` mais il est un module parmi des dizaines Java-first — pas un outil TS natif
- Le pattern Seed4J est très propre : chaque module = 1 factory (domain) + 1 service (application) + 1 configuration Spring (infrastructure/primary) — transposable en TS avec le même découpage
- La terminologie de Seed4J (ModuleFactory, ModuleResource, ModuleProperties...) est trop proche pour un projet distinct — nécessité de renommer entièrement
- L'architecture hexagonale garantit que le domaine (BrickBuilder, etc.) est testable sans Next.js, sans filesystem — pure TypeScript

## Décisions prises
- **Stack outil** : Next.js fullstack (App Router) + TypeScript 5 + vitest
- **Bounded contexts** : `brick/` (moteur générique) et `catalog/` (implémentations concrètes par technologie)
- **Terminologie** : Brick / BrickBuilder / BrickContext / Patch / BrickRegistry / ProjectWriter / BrickDefinition
- **Versioning** : version de l'outil généré (ex: Express 4.x vs 5.x), pas versioning du module lui-même
- **Compatibilité inter-briques** : reportée à une phase ultérieure, construire module par module d'abord
- **Architecture générée** : brique dédiée `hexagonal-structure` pour les projets qui veulent l'archi hexa — optionnelle, pas implicite
- **Approche progressive** : pas de liste de briques cibles fixée, ajout au fil de l'eau
