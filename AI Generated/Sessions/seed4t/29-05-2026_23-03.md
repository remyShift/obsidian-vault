## Decouvertes
- Confirmation que le projet n'a pas besoin de base de données : le catalogue est statique (InMemoryBrickRegistry), l'app est stateless
- Prise de conscience importante : un plan TDD ne doit PAS pré-designer les abstractions (ports, Value Objects) — elles doivent émerger du refactor, pas être imposées dès le départ
- L'archi hexagonale n'est pas un point de départ TDD mais un résultat : chaque port émerge quand un test révèle une dépendance problématique (ex: `ProjectWriter` émerge quand le test touche le filesystem)

## Decisions prises
- Documents et skills entièrement en anglais (CLAUDE.md, add-brick.md, tdd-cycle.md, hexagonal-architecture.md)
- Pas de base de données dans le projet — catalogue hard-codé, données transientes
- Approche TDD naive-first : premier test avec des strings brutes, les types/classes émergent au refactor uniquement quand c'est justifié
- Première brique ciblée : `typescript` (la plus simple et isolée)
- Séquence d'émergence validée :
  1. `Patch`, `NpmDependency`, `BrickFile` ← refactor test 1 (nommer le retour)
