## Problèmes / Bugs
- Le bloc `describe('has(key)')` a été écrasé lors de l'écriture des tests circular deps (branches séparées, pas de merge — comportement normal mais à surveiller)
- Vitest affichait 17 tests au lieu de 20 attendus : cause identifiée (tests `has()` absents de la branche circular deps)

## Découvertes
- La lib ne détecte pas les cycles entre providers lazy : `inject(A)` → `inject(B)` → `inject(A)` = stack overflow sans message utile
- La solution est un `Set resolving` dans la closure de `piqure()` : coût minimal, zéro changement dans `Value.ts`
- Si le provider throw pour une raison externe, `resolving.delete(key)` doit être dans un `finally` pour ne pas polluer les appels suivants
- `piqure()` utilise `Map.has()` en interne, ce qui rend `has()` trivial a implémenter (une ligne)
- Les tests TDD indirects (cycle A → B → A) peuvent passer sans nouveau code si le cycle direct est déjà géré — c'est normal et attendu selon les règles TDD
- `key<T | undefined>` ne distingue pas "non enregistré" de "enregistré avec undefined" : ambiguité sémantique que `has()` évite
- `--force-with-lease` est plus safe que `--force` : refuse si quelqu'un a pushé entre temps sur la branche

## Décisions prises
- Deux contributions séparées sur deux branches distinctes : `has()` et circular dep detection
- Ne pas modifier `key()` pour utiliser `Symbol.for()` globalement : trop de perte d'isolation. `keyFor()` existe pour les cas RSC
- Approche TDD stricte : RED visible (stack overflow ou erreur manquante) avant tout code de production
- Maintenir `has()` face au reviewer : le pattern `key<T | undefined>` déplace l'optionalité dans les types et force tous les consommateurs à gérer `undefined`
