## Découvertes
- **Le SKU est la clé de jointure avec BigBlue** : `apps/server` lit l'inventaire BigBlue et matche par SKU (`inventoryMap[sku]` dans `xmlController.js`, via `getAllInventoryFromBigblue()`). Aussi clé sur order items + bundles.
- **Le repo ne pousse JAMAIS de produits/SKU vers BigBlue** : intégration read-only (inventaire) + webhooks de statut de commande + push de bundles uniquement.
- **Aucun signal first-class "ce SKU est live sur BigBlue"** ; seuls proxys : `status` (`Live`/`Staged`/`Offline`) et `syncMetadata.lastSyncedAt`.
- `syncMetadata` = sync vers la **legacy MongoDB**, pas BigBlue (endpoint `/sync/product/:id`).
- **Pas de RBAC** : collection `Users` en auth nu (aucun champ custom), tous les users authentifiés = admin. Pattern `roles` documenté dans `apps/cms/AGENTS.md` mais non implémenté.
- Patterns réutilisables déjà présents : `syncButton` (`type: 'ui'` + composant `SyncProductButton`) et `CreateOnlyTextField` sur l'EAN → précédents directs pour une action manuelle et un champ conditionnel.
- La PR `feat/edits-product-picker-admin-label` déplace déjà `useAsTitle` de `sku` → `adminLabel` (champ caché, calculé) — utile pour le fix duplicate.
- Le vrai défaut de design : le gel du SKU se déclenche sur "une valeur existe" au lieu de "le produit est engagé downstream (BigBlue)".
- **Bulk-add** : `Edits.products` (`hasMany`) existe déjà → la feature ne nécessite aucun changement de schéma. Tous les slots `admin.components` de la list view (`beforeList`, `Description`, `listMenuItems`, `beforeListTable`, `afterListTable`) sont rendus **dans** le `SelectionProvider` → `useSelection()` marche partout. **Aucun slot public** dans la barre native "X selected / Edit / Delete".
- Bulk-edit natif (`EditMany/DrawerContent.js`) = `PATCH` qui **remplace** le champ + force `_status:published` → ne sait pas append. Le vrai pivot du choix = **append vs replace**, pas la direction de la relation.

Édition de l'appartenance depuis le côté relationship uniquement. `allowCreate` ne **crée** qu'un nouveau doc lié, n'**ajoute pas** un existant, et n'est pas bulk → **ne peut pas** faire le bulk-add. - `Products.status` (Live/Staged/Offline) : aucun `readOnly`/`access` → n'importe qui change la visibilité storefront en un clic.
- **PostHog flags = client-side, end-user-facing** (évalués via `useActiveFeatureFlags`, aucune eval server-side) → couche **distincte** de l'accès admin Payload. "FF côté Payload" = contrôle d'accès admin, **pas** un flag PostHog (ne pas confondre dans le doc).

## Décisions prises
- **Slug — il faut les DEUX chemins, pas un seul** (correction d'un faux départ) : (1) override checkbox = hook `async` explicite (`generateProductSlugHook`/`generateSubcategorySlugHook`), que Payload `await` → save serveur ; (2) arg `slugify` → bouton client (le `slugifyHandler` l'`await`). Les deux délèguent à la même fonction `slugifyProduct`/`slugifySubcategory`. **L'override N'EST PAS redondant** : le hook `generateSlug` par défaut de Payload assigne le slugify SANS `await` → une Promise async atterrit dans `data.slug` → Mongoose rejette (cassait les tests d'intégration). Premier essai (override supprimé, tout sur l'arg) = erreur.
- **Migration subcategory NON découplée** : elle importe le slugify corrigé, donc un env reconstruit from scratch produira des slugs corrects (prod inchangée, déjà migrée). Option de figer la migration sur l'ancien comportement laissée ouverte à Rémy.
- **Test subcategory pré-existant** : assertion `vitamin-c--e` corrigée en `vitamin-c-e` (règle "les tests vérifient le comportement attendu, pas le cassé"). Aucun nouveau test ajouté ; le test produit a été supprimé par Rémy.
- **RFC RBAC** : doc = RFC technique, profondeur "case + direction" (design complet → ticket dédié plus tard). Direction recommandée = modèle de rôles **minimal** (champ `roles` sur `Users` : `select` `hasMany` `saveToJWT`) + helpers `apps/cms/src/access/` (`isAdmin`, `hasRole`) réutilisés sur collection/field access **et** endpoints. Migration **incrémentale** : étape 1 = ajouter `roles`, tout le monde admin par défaut → **zéro lock-out**. Pilote proposé = champ `slug`.
- **Cadre honnête assumé dans la RFC** : valeur = sûreté opérationnelle + travail par étapes, **pas** sécurité anti-malveillant (équipe de confiance) ; si pas d'éditeur non-tech bientôt → ne faire que les étapes 1-3. Pas de matrice de permissions (YAGNI).
- **Livraison RFC** = markdown rédigé, Rémy colle dans Notion (pas de publication via MCP).
