## Découvertes
- **Le SKU est la clé de jointure avec BigBlue** : `apps/server` lit l'inventaire BigBlue et matche par SKU (`inventoryMap[sku]` dans `xmlController.js`, via `getAllInventoryFromBigblue()`). Aussi clé sur order items + bundles.
- **Le repo ne pousse JAMAIS de produits/SKU vers BigBlue** : intégration read-only (inventaire) + webhooks de statut de commande + push de bundles uniquement.
- **Aucun signal first-class "ce SKU est live sur BigBlue"** ; seuls proxys : `status` (`Live`/`Staged`/`Offline`) et `syncMetadata.lastSyncedAt`.
- `syncMetadata` = sync vers la **legacy MongoDB**, pas BigBlue (endpoint `/sync/product/:id`).
- **Pas de RBAC** : collection `Users` en auth nu (aucun champ custom), tous les users authentifiés = admin. Pattern `roles` documenté dans `apps/cms/AGENTS.md` mais non implémenté.
- Patterns réutilisables déjà présents : `syncButton` (`type: 'ui'` + composant `SyncProductButton`) et `CreateOnlyTextField` sur l'EAN → précédents directs pour une action manuelle et un champ conditionnel.
- La PR `feat/edits-product-picker-admin-label` déplace déjà `useAsTitle` de `sku` → `adminLabel` (champ caché, calculé) — utile pour le fix duplicate.
- Le vrai défaut de design : le gel du SKU se déclenche sur "une valeur existe" au lieu de "le produit est engagé downstream (BigBlue)".
- **Bulk-add** : `Edits.products` (`hasMany`) existe déjà → la feature ne nécessite aucun changement de schéma. Tous les slots `admin.components` de la list view (`beforeList`, `Description`, `listMenuItems`, `beforeListTable`, `afterListTable`) sont rendus **dans** le `SelectionProvider` → `useSelection()` marche partout. **Aucun slot public** dans la barre native "X selected / Edit / Delete".
- Bulk-edit natif (`EditMany/DrawerContent.js`) = `PATCH` qui **remplace** le champ + force `_status:published` → ne sait pas append. Le vrai pivot du choix = **append vs replace**, pas la direction de la relation.
- **`join` field** = virtuel ("no new data is stored"), `access: { create: never, update: never }` → **non écrivable**. Édition de l'appartenance depuis le côté relationship uniquement. `allowCreate` ne **crée** qu'un nouveau doc lié, n'**ajoute pas** un existant, et n'est pas bulk → **ne peut pas** faire le bulk-add. Sa vraie valeur = visibilité inverse (Produit → ses Edits) sans duplication = réponse idiomatique au smell d'origine + tue le dernier pro de l'Approche A.
- Périmètre de ce ticket = **cadrer le problème**, pas trancher la solution (décision en meeting).
- Reco : **découpler les deux besoins** — Option D (adminLabel + clear SKU au duplicate) pour le duplicate ; Option B (action manuelle, confirmée, downstream-aware, persistée via Payload) pour la correction ; Option C (RBAC / unlock SKU pour tech) **différée** comme chantier séparé.
- Contexte établi : **c'est l'équipe contenu / client qui édite les produits** → un bouton "régénérer" ouvert à tous est risqué, le gating par rôle a du sens à terme.
- Brief livré en anglais (thread Michele/Diego en EN).
- **Bulk-add** : Approche B retenue comme corps du plan (composant + endpoint `POST /api/edits/:id/add-products`, dedupe+merge, `overrideAccess:false`+`user`, préserver `_status`). Picker cible = `useListDrawer` sur `edits`. Placement bouton = `beforeListTable` (reco) ou `listMenuItems` (swap 1 ligne).
- **`join`** = gardé mais **découplé** : feature sœur "visibilité inverse" hors scope du bulk-add (pas un substitut à l'écriture), avec porte de décision (ajouter seulement si l'équipe contenu en a besoin, sinon YAGNI).
- **Top banner — le message free-shipping existe DÉJÀ** mais dans le chemin fallback hardcodé (next-intl `navbar.messages.freeShippingMessage` / react-i18next `topBarMessage2`). La banner a une **double source** pilotée par le flag `isDevAnnouncementBarEnabled` : flag ON + CMS rempli → segments CMS (texte/lien brut, **aucune valeur dynamique**) ; flag OFF → fallback hardcodé (montant dynamique). La demande = **fusionner les deux**.
- Seuil résolu **côté client** : web via `useFreeShippingThreshold()` (number, par pays + service sélectionné, défaut 70) ; web_client via `selectedService?.freeshipping`. La valeur dépend d'atoms Jotai (service + pays) → **contexte client**.
- **Payload n'a aucun built-in de templating/merge-tags** sur une valeur de champ. Conséquence : ni stockage en CMS (valeur dynamique par pays/service, duplique le Mongo `delivery_service.freeshipping`), ni hook `afterRead` (contexte client absent au read + global ISR-caché 3600s) → l'injection **doit** se faire au render, côté front.
- Global `announcement-bar` = `messages[]` → `segments[]` (type text|link), champ `text` localisé en plain text (pas de richText).
- **Payload 3.84 expose `slugField` nativement** (`payload/dist/fields/baseFields/slug`). Slugify par défaut = `val.trim().replace(/ /g,'-').replace(/[^\w-]+/g,'').toLowerCase()` (pas de collapse, pas d'accents).
- `slugifyHandler` (`@payloadcms/next`) = la pièce clé : c'est ce que le bouton client appelle, et il lit `field.custom.slugify` (alimenté par l'arg `slugify`), **pas** le hook checkbox override. Le hook checkbox par défaut `generateSlug` appelle aussi le même arg → en passant l'arg, l'override devient inutile (un seul générateur).
- `fieldToUse` est **déprécié** → `useAsSlug`.
