## Problèmes / Bugs
- `url?: string | null` sur toutes les collections Media (Media, LogoMedia, ProductMedia) — Payload génère ce champ comme optionnel par design même quand un upload est forcément présent en production
- `computeCartSnapshot.ts` duplique la logique de narrowing de `product.ts` avec des casts inline `as Brand` / `as ProductMedia` au lieu d'utiliser des guards propres
- `typeSafeDepth: true` n'existe pas dans les types de config Payload 3.79.1 — la PR est toujours ouverte

## Découvertes
- `typeSafeDepth` (PR #9782) : toujours ouverte, pas mergée, pas disponible sur aucune version — était une fausse piste
- Payload SDK (`@payloadcms/sdk`, PR #9463) : mergée et disponible — résout `string | T` pour les consommateurs REST uniquement, pas pour la Local API ni les hooks CMS
- Il n'existe pas de solution officielle pour fixer les types générés par Payload (`url?: string | null`, `string | T`) — c'est une limite fondamentale du générateur de types
- La communauté Payload a convergé universellement vers le pattern `deref`/`assertPopulated` — c'est le standard de facto, pas un workaround maison
- Trois patterns communautaires documentés : `deref` function (le plus répandu), `RemoveStringType` utility type, script post-génération regex (fragile, abandonné)
- `packages/shared` existe déjà et est consommé par `apps/web` ET `apps/cms` — c'est le bon endroit pour centraliser les guards

## Décisions prises
- Approche retenue : Option A — créer `packages/shared/src/payload/guards.ts` avec `isPopulated`, `assertPopulated`, `WithUrl<T>`, `assertUrl` — migration des deux fichiers pilotes
- Payload SDK (Option B) : ne couvre pas `computeCartSnapshot.ts` (Local API), scope trop large pour le PoC actuel
- Option C (mapper boundary) : convention complémentaire à A, pas exclusive
- Plan mis à jour pour refléter la réalité : `typeSafeDepth` retiré, Payload SDK documenté avec ses vraies limites
- Accepté que le problème ne peut pas être réglé à la source — seule standardisation du workaround est possible
