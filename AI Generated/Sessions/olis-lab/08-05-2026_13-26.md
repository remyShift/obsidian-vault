## Problèmes / Bugs
- `url?: string | null` sur toutes les collections Media — Payload génère ce champ comme optionnel par design
- `computeCartSnapshot.ts` duplique la logique de narrowing de `product.ts` avec des casts inline
- `typeSafeDepth: true` n'existe pas dans les types de config Payload 3.79.1 — PR #9782 toujours ouverte
- Payload SDK (PR #9463) : "type safe populate" signifie probablement que les options sont typées en entrée, pas que le type de retour change — à vérifier avant d'inclure comme solution
- SearchPage : `handleAddToBag` cherchait les bundles dans `rawProducts` → `undefined` → click silencieux sans effet
- SearchPage : prop `combinedProducts` passée à `ProductResults` qui attend `products` — erreur TypeScript

## Découvertes
- `typeSafeDepth` (PR #9782) : toujours ouverte, pas mergée, fausse piste
- Payload SDK : collègue avait déjà essayé populate depuis le source code sans succès — renforce le doute sur l'utilité pour `string | T`
- Il n'existe pas de solution officielle pour fixer les types générés par Payload — limite fondamentale
- La communauté Payload converge universellement vers le pattern `deref`/`assertPopulated` — standard de facto
- Trois patterns communautaires : `deref` function (le plus répandu), `RemoveStringType` utility type, script post-génération regex (fragile, abandonné)
- `flatMap` avec `[]` = map + filter en une passe : élément qui retourne `[]` disparaît du résultat
- `QuickAddToBagPopup` gère nativement les bundles : branche `isBundle` dans `checkAvailability` appelle `bundlesApi.checkBundleAvailability`, affiche "What's included"
- `processingProductIdAtom` est reset à `null` dans le `onClose` de `Routes.tsx` — le guard `if (processingProductId) return` dans `ProductCard` se reset correctement à la fermeture du popup
- `BundleSearchResult` et `ShopProduct` sont des types distincts — le cast `as unknown as ShopProduct` est nécessaire pour passer un bundle au popup, les champs requis (`isBundle`, `imgUrl`, `price`, `products`, `sku`, `slug`) sont tous présents dans `BundleSearchResult`

## Décisions prises
- Approche retenue : Option A — centraliser dans `packages/shared/src/payload/guards.ts`
- 4 utilities : `isPopulated` (type guard non-throwing), `assertPopulated` (throw), `WithUrl<T>` (type), `assertUrl` (throw)
- `assertPopulated` pour les mappers (depth garanti), `isPopulated` pour les hooks où le depth n'est pas contrôlé
- Option C (mapper boundary) : convention complémentaire à A
- Payload SDK retiré du plan comme solution — doute sur l'utilité réelle, scope trop large
- Accepté que le problème ne peut pas être réglé à la source
- SearchPage : pas de hook intermédiaire (`useSearchPageResults`) — combining logic inline dans `SearchPage.tsx`, plus simple
- SearchPage : bundles → `QuickAddToBagPopup` (pas navigate) — cohérent avec shopAll, permet l'ajout au panier directement depuis la recherche
- `mapBundleSearchResult` locale au module `SearchPage.tsx` (pas extraite dans `utils/mapProducts.ts`) pour cette étape
