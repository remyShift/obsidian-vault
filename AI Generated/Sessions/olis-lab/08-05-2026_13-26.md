## Problèmes / Bugs
- `url?: string | null` sur toutes les collections Media (Media, LogoMedia, ProductMedia) — Payload génère ce champ comme optionnel par design même quand un upload est forcément présent en production
- `computeCartSnapshot.ts` duplique la logique de narrowing de `product.ts` avec des casts inline `as Brand` / `as ProductMedia` au lieu d'utiliser des guards propres

## Découvertes
- Payload 3.79.1 (version utilisée dans le projet) inclut `typescript.typeSafeDepth: true` — feature opt-in qui génère des types génériques depth-aware pour les relations
- La feature `typeSafeDepth` ne résout PAS le problème `url?: string | null` sur les Media — problème orthogonal
- `packages/shared` existe déjà et est consommé par `apps/web` ET `apps/cms` — c'est le bon endroit pour centraliser les guards
- La collection `Config` dans `payload-types.ts` liste toutes les collections avec leur slug exact — utile pour typer les resolvers génériques
- Memory LyonCraft était complètement périmée (encore "AI generative + Monsters Inc") — le vrai talk est sur le parcours vendeur -> dev
- Agent /evolve avait identifié le mauvais dernier rapport (25-04 au lieu de 01-05) — à surveiller pour les prochains runs

## Décisions prises
- Approche recommandée court terme : créer `packages/shared/src/payload/guards.ts` avec `isPopulated`, `assertPopulated`, `WithUrl<T>`, `assertUrl` — migration des deux fichiers pilotes
- `typeSafeDepth: true` : ne pas activer directement en prod, faire un spike sur une branche dédiée pour mesurer l'impact (nombre de fichiers cassés)
- L'idée d'un resolver discriminant par collection est valide conceptuellement mais `isPopulated<T>` générique couvre le même besoin sans duck typing fragile
