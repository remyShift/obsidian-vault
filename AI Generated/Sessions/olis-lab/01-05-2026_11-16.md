## Problemes / Bugs
- `@payload-enchants/seo@1.2.2` compile avec Payload `3.0.0-beta.61` → incompatible avec Payload `3.79.1` stable
- `useFieldProps` supprime de `@payloadcms/ui` en stable, remplace par `useFieldPath` (retourne une `string` directe, plus un objet `{ path }`)
- `Cannot find module '@payload-enchants/seo/dist/types'` : chemin non expose dans l'`exports` map du package, inaccessible avec `moduleResolution: bundler`
- `fields` dans le nouveau plugin est un tableau statique (AJOUTE des champs) et non une fonction de filtrage comme dans l'ancien plugin → impossible de masquer le champ `preview` de la meme facon
- `seoPlugin` n'est plus le nom d'export : le package exporte `seo` uniquement

## Decouvertes
- `@payload-enchants/seo@1.2.2` a ete publie avec des peer deps pointant vers `3.0.0-beta.61`, ce qui le rend incompatible avec toute version stable de Payload 3.x
- `useFieldPath` (nouveau) retourne directement une `string`, contrairement a `useFieldProps` (ancien) qui retournait `{ path: string }`
- L'option `fields` du nouveau plugin fait un spread (`...pluginConfig?.fields || []`) : elle ajoute des champs supplementaires, pas de remplacement/filtrage des champs par defaut
- `pnpm patch` est le mecanisme standard pour patcher un package tiers sans le forker : cree un fichier `.patch` versionne dans le repo, applique automatiquement a chaque `pnpm install`
- Message envoye au mainteneur du plugin pour signaler l'incompatibilite Payload stable

## Decisions prises
- Rester sur `@payloadcms/plugin-seo` officiel : plus stable, zero patch, l'integration OpenAI se fait via `generateTitle`/`generateDescription` cote serveur
- Abandonner `@payload-enchants/seo` : publie pour Payload beta, incompatible en production, necessite un patch fragile
- Pas de composant custom ni d'endpoint custom : les boutons natifs du plugin par champ suffisent
- Le feedback UX (toast) est sacrifie : si OpenAI echoue, le champ se vide silencieusement (limite connue du plugin natif, acceptable pour l'instant)
