## Découvertes
- **CRA → Payload : chemin déjà existant** : `cmsClient` (PayloadSDK) dans `apps/web_client/src/service/cms/cmsClient.ts`, lecture de global via `cmsClient.findGlobal({slug})` + `assertX` depuis `@olis-lab/shared/payload`. `useTradingPlanQuery.ts` = template.
- La navbar shop du CRA dérive 6 colonnes (skin/body/skinType via API catégories Express + skinConcern/edits/tools statiques). `skinType.path` = `/skintypes/<slug>` (Express `generateSlug`).
- Le `type: 'row'` Payload est purement présentationnel : les champs restent au même niveau de données → `siblingData` reste accessible dans `validate`, et les types générés ne changent pas.
- Les hooks catégories (`useCategoriesQuery`/`useSubcategoriesQuery`/`useSkinTypes`) sont utilisés ailleurs (CategoriesPage, CollectionPage, RecommendedForMe, SubCategoriesSection) → ne pas supprimer.
- Localisation d'arrays Payload : pour seeder EN+FR sans perdre l'EN, il faut écrire EN → relire les ids → réécrire FR avec les ids (sinon recréation des rows).
- `FooterTrustBar.tsx` (apps/web) = la **référence Embla qui marche dans Next** : même config (`axis:'y'`, `loop`, `Autoplay`, `watchDrag:false`) mais DOM minimal, sans CSS Embla externe. À copier pour tout carousel maison côté Next.
- `ExternalLink` de `typography.tsx` = simple wrapper `(props) => <a {...props} />` (pas de style/target par défaut).
- Le typage générique camelCase porté du `useFeatureFlags` CRA est jugé trop complexe en review pour côté Next : préférer un retour à clés explicites tant qu'il y a peu de flags.
- **Payload : une relation pointant vers un document supprimé n'est PAS peuplée même à `depth >= 1`** — Payload renvoie l'ID brut (string) au lieu de l'objet. Tout code qui suppose « depth=1 ⇒ objet peuplé » doit gérer le cas string (relation orpheline).
- **Convention déjà en place dans le codebase** : `products.ts` (`resolveRelationNames`, `resolveSubcategoryCategories`) **filtre silencieusement** les relations orphelines via `flatMap`. `edits.ts` diverge volontairement (throw « fail loudly ») — c'est cette divergence qui casse la prod.
- Coupling import : `schemas/edits.ts` importe `STATUS_OPTIONS` depuis `collections/Products.ts`, qui tire toute la chaîne jusqu'à `payload.config.ts` (S3) → un test unitaire du transformer ne peut pas être chargé sans env/`vitest.setup.ts` (dynamic import). `curationSettings.spec.ts` y échappe en n'important pas les schemas.
