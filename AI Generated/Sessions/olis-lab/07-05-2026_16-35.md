## Découvertes
- Le linter du projet a déplacé la validation Joi de `middleware/validateInput.js` vers `validations/xmlValidation.js` — le fichier `validateInput.js` reste vide
- `#` (private class fields JS natif) supporté depuis Node.js v14.6 — utilisé pour `#fetchLiveProductsWithInventory`
- `Promise.all` sur le fetch produits + BigBlue réduit le temps de génération de feed : les deux appels étaient séquentiels avant
- `CategoriesFilter` (2440 lignes) accepte `ShopProduct[]` et gère déjà les bundles via `isBundle: true` — un seul tableau suffit pour filtrer produits + bundles
- `BundleSearchResult` (SearchPage) et `ShopProduct` sont des types différents — un adapter `bundleSearchResultToShopProduct` est nécessaire avant de passer à `CategoriesFilter`
- `BundleSearchResult` n'a probablement pas de `concerns`/`skinTypes`/`subcategories` dans ses translations → ces filtres seront vides pour les bundles (attendu)

## Décisions prises
- `#fetchLiveProductsWithInventory(fields)` : méthode privée dans la classe (pas module-level), appelée via `XmlController.#fetchLiveProductsWithInventory()`
- Validation langue via middleware Joi plutôt qu'inline dans chaque méthode — un seul point de vérité
- Klaviyo feed : `sale_price>n/a</sale_price>` si pas de discount (Klaviyo exige que tous les champs soient présents sur chaque item pour le mapping automatique)
- Clé S3 Klaviyo : `feeds/klaviyo-${lang}.xml` (cohérent avec `feeds/gmc-${lang}.xml`)
- SearchPage : vue combinée produits + bundles dans un seul onglet avec `CategoriesFilter` en sidebar — onglet "bundles" séparé supprimé
- Adapter `bundleSearchResultToShopProduct` à créer dans `src/utils/` pour convertir les bundles du search en `ShopProduct` compatible
