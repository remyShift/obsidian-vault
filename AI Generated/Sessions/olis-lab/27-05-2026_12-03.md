## Problemes / Bugs
- Le `Write` initial a ecrit dans `collections/TradingPlan.ts` au lieu de `collections/globals/TradingPlan.ts` — detecte via `ls`, corrige en travaillant directement sur le bon fichier
- Premiere version utilisait `virtual: true` (champs non stockes en DB) — corrige apres clarification de Remy : les champs doivent etre persistes
- Le formatter du projet convertit systematiquement les apostrophes ASCII en guillemets typographiques (U+2018/U+2019) lors des edits — casse TypeScript et esbuild
- La comparaison dans `guardProduct.ts` utilisait un apostrophe typographique depuis sa creation — le test Collectibles echouait silencieusement (vitest ne type-check pas, esbuild ignore)
- `ProductMedia`, `Brand`, `SkinType`, `Concern`, `ProductActive` ont tous des champs requis non triviaux — les fixtures `{ id: '...' }` ne suffisent pas pour TypeScript strict
- Limitation native Payload identifiee : le champ `relationship` utilise `useAsTitle` a la fois pour l'affichage et la recherche — impossible de les dissocier sans custom component

## Decouvertes
- `virtual: true` sur un champ Payload = champ calcule a la volee via `afterRead`, non stocke en DB
- `@payloadcms/sdk` supporte `findGlobal()` pour fetcher un Global Payload depuis le navigateur
- `apps/web` utilise deja `@payloadcms/sdk` avec `@olis-lab/shared/cms` — pattern reutilisable tel quel
- Le type `TradingPlan` est deja genere dans `packages/shared/src/payload-types.ts:1048`
- `REACT_APP_PAYLOAD_CMS_URL` est configure en local et staging mais absent de `.env.prod`
- Le formatter convertit uniquement les nouvelles additions (pas le contenu pre-existant)
- `tsc --noEmit` revele des erreurs que vitest (esbuild) ignore silencieusement — a lancer regulierement
- Tester des etats impossibles selon les types au niveau `assertProduct` est non-pertinent : ces cas appartiennent aux helpers bas niveau (`guard.test.ts`)
- Le champ `relationship` Payload ne supporte pas nativement la recherche multi-champs : `useAsTitle` est a la fois le label affiche et le champ sur lequel porte la recherche
- Option A (champ denormalise `adminLabel` persiste en DB) couvre brand + title + sku + subcategory sans custom component
- Option B (custom field component via `admin.components.Field`) donne un controle total mais necessite ~2-4h de dev

## Decisions prises
- `title` et `ctaLink` de `brandOfTheMonth` : stockes en DB (pas `virtual`), remplis automatiquement via hook `beforeChange` global
- Pattern SDK pour `web_client` : calque exact sur `apps/web/lib/api/cms.ts`
- Hook `useTradingPlanQuery` : `queryKey` inclut la locale, `staleTime: 5min`
- Appel du hook une seule fois par page (`Shop.tsx`, `Home.tsx`), slices passes en props aux sous-composants
- Tests `assertProduct` couvrent uniquement le contrat de la fonction : unions Payload et logique metier (`howToUse` conditionnel, renommage `skinTypes`)
- Pas de `makeRawProduct` ni de `as unknown as Product` dans les tests
- Strings contenant apostrophes = guillemets doubles (regle de code active dans ce projet)
- Product picker Edits : approche recommandee = Option A (champ `adminLabel` denormalise) pour un gain immediat ; Option B (custom component) si insuffisant apres usage
