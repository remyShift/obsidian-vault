## Découvertes
- Claude Code écrit automatiquement les permissions dans le `.claude/settings.local.json` du projet courant quand on clique "Always allow" pendant une session — le fichier cible n'est pas le settings global, il faut le déplacer manuellement si la permission est générique
- Les permissions MCP lean-ctx, obsidian, WebSearch ne sont pas spécifiques à un projet et doivent vivre dans `~/.claude/settings.json`
- Pour résoudre les conflits d'une PR sans polluer le diff : cherry-pick des commits originaux sur develop (pas de merge) — le merge aspire toute l'historique de develop dans la branche
- Un fichier peut avoir des erreurs TypeScript post-merge sans avoir de markers de conflit git (auto-merge silencieux d'un import vers un module supprimé)
- `useSearchPageQuery` peut retourner plusieurs sous-queries dans un seul hook pour centraliser la logique de search sans dépendances circulaires
- `Trans` component i18next permet d'injecter des balises HTML dans les traductions sans unsafe HTML (`components: { query: <strong /> }`)
- La branche `feat/search-page-others-collections` a déjà les types `ActiveSearchResult` et `BrandSearchResult` dans `types.ts` et `getActivesSearch` dans `activesApi.ts` — pas encore de `getSearchBrands` dans `brandsApi.ts`

## Décisions prises
- Toutes les permissions génériques (outils MCP transverses, WebSearch, WebFetch sur docs officielles) vont dans le settings global, pas dans les settings projet
- Stratégie de résolution de conflits : toujours cherry-pick ou rebase, jamais merge, pour garder le diff de PR propre
- Ne pas ajouter de co-auteur Claude dans les commits sans demande explicite
- Debounce 500ms conservé sur la search (live search, pas de suggestions) — `showResults` à partir de 3 chars (pas 2)
- `SearchTabs` stateless avec type `SearchTab` exporté — extensible sans modifier le composant
- Feature flag `dev_search` avec `LOCAL_DEV_MOCKS: true` pour activer la page en dev sans PostHog
