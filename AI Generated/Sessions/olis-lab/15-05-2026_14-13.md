## Découvertes
- `computeCartSnapshot` : unique hook beforeChange du projet, logique critique et subtile notamment le check `'discount' in data` (vs `?.percentage ?? fallback`) qui évite de réutiliser une remise supprimée
- `transformPayloadProductToLegacy` et `transformPayloadBrandToLegacy` : fonctions pures, testables sans infrastructure — priorité haute car corruption silencieuse en legacy DB si bug
- `uniquePerCategory` : contrainte d'unicité composite implémentée manuellement via requête DB (Payload ne supporte pas les unique composites nativement)
- Payload ne recommande pas `mongodb-memory-server` — leur adapter Mongoose a un lifecycle interne (init plugins, enregistrement collections, index) qui présuppose une vraie connexion
- Pattern Payload recommandé : DB dédiée aux tests (`olis_lab_test`) via env var, même instance MongoDB locale, cleanup par tracking d'IDs en `afterEach`
- Seed via Local API directement (`payload.create(...)`) — pas de raw MongoDB, pas de migration files

## Décisions prises
- `mongodb-memory-server` écarté pour les tests d'intégration Payload
- DB de test = dedicated MongoDB database (`mongodb://127.0.0.1/olis_lab_test`) via `DATABASE_URL` env var
- Cleanup pattern = tracking IDs + `afterEach` delete (pas de flush complet)
- Seed = fonctions locales via Local API, une par suite, auto-contenue
- Factories = pas nécessaires pour l'instant (27 scénarios, 2 suites d'intégration)
- Tests unit (transformateurs, hook avec req.payload mocké) = aucune DB requise
- E2E Playwright = déprioritisé (trop fragile pour la valeur)
