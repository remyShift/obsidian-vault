## Découvertes
- Zed a une **Rules Library** native (palette commandes > "agent: open rules library") — équivalent system prompt persistant sans provider externe
- Les règles projet se placent dans `.zed/rules/project.md` au root du repo (équivalent CLAUDE.md)
- Payload **blocks** = union typée en tableau JSON, drag-and-drop dans l'admin, chaque item peut avoir une structure différente (contrairement à `array` où tous les items ont la même structure)
- Payload **Globals** = singleton, une seule entrée, parfait pour homepage/shop page — `globals: []` actuellement vide dans le projet
- La distinction clé blocks vs champs groupés : les blocks ne sont utiles que si Olivia a besoin de réordonner ou ajouter/supprimer des sections ; pour juste éditer le contenu dans une structure fixe, un Global plat suffit
- `CategoriesFilter` lit tout son filtering depuis `product.translations[lang].*` (`skinTypes`, `subcategories`, `concerns`) et non des champs top-level — ces données sont déjà dans le `.select()` du backend, donc aucun changement backend nécessaire pour le filtering de la SearchPage
- `SearchResultItem` avec un champ discriminant `type: 'product' | 'bundle'` est redondant puisque `ShopProduct` a déjà `isBundle?: boolean` — utiliser `ShopProduct` directement est plus propre
