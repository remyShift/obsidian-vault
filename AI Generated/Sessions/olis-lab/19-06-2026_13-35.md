## Découvertes
- Payload ne génère pas de type `SiblingData` dédié par champ (siblingData est contextuel), mais exporte des génériques `Validate`/`Condition`/`FieldHook` avec un slot `TSiblingData` qu'on alimente avec le type généré. Ordre du générique : `Validate<TValue, TData, TSiblingData, TField>`.
- Distinction produits côté query : `status: 'Live'` = champ métier legacy (select Live/Staged/Offline, synchro Notion, c'est lui qu'on filtre pour le storefront) vs `_status: 'published'` = flag draft/publish interne Payload, géré automatiquement par le param `draft` du SDK. Les deux sont orthogonaux.
- `labelFor` dans `Navbar.tsx` a deux sources selon feature flag : label Payload (CMS) ou data statique legacy, d'où le fallback `navItemLabels?.[id] ?? t(labelKey)`. Le cast `as NavItemType` est sauvé par le garde `hasSubmenu` (brands n'a pas de submenu).
- Convention de nommage handlers vs props (règle Kyle) : le préfixe `on` est réservé aux **props** (`onClick`, `onDismiss`...) ; la fonction qu'on définit puis passe se nomme `handleXxx`. Ex : `<Component onDismiss={handleDismiss} />`.
- `apps/cms` consomme `@olis-lab/shared` via le **dist buildé** (`exports["./payload"] → ./dist/payload/index.js`), pas le src ; aucun alias tsconfig ne redirige vers src. Toute modif du barrel src exige un rebuild de `shared` avant de tester en local. Le `dist` est gitignoré (artefact `tsc`).
- En CI, le job `cms-int-tests` exécute `Build Base Packages` (`turbo run build --filter=@olis-lab/shared`) avant `pnpm test:int` → le dist y est toujours frais, donc la CI ne casse jamais pour la cause « dist périmé » (problème purement local).
- `legacyId` = ObjectId Mongo legacy (24 hex), posé par la sync (`endpoints/sync/products.ts`, `newId.toString()`), sert de `_id` au snapshot cart. Un produit créé purement dans Payload n'en a pas tant qu'il n'est pas synchronisé.
- Déclenchement du hook `computeCartSnapshot` : `beforeChange` sur Products (`Products.ts:95`), donc appelé sur **create ET update** (admin, `payload.update`, sync, GraphQL/REST). 3 garde-fous : `req.context.skipCartSnapshot`, `data._status !== 'published'`, et la branche `operation === 'update' && originalDoc`. Net : snapshot écrit **uniquement sur update d'un produit publié sans skip context**. Le create est un `// TODO` (no-op). D'où : un produit neuf n'a pas de snapshot tant qu'il n'a pas reçu un premier update publié (dans la factory, c'est l'update fr qui l'écrit).
- Contradiction interne hook/schéma : le hook remplit les locales manquantes avec `?? ''` (il tolère), le schéma `cartProductSchema` rejette `''` (`min(1)`, il exige). Tranché en actant la règle métier « snapshot bilingue ».

## Décisions prises
- Typer siblingData via les génériques Payload + le type généré, jamais de type inline. Pattern aligné sur AnnouncementBar.
- Split du validateur en deux fonctions dédiées plutôt que via un flag `internalOnly` (décision du call avec Kyle).
- La nav top-level reste statique (noms et ordre figés) pour cette PR ; rendre éditable et réorderable = chantier futur, hors scope.
- staleTime navbar à 24h car contenu éditorial qui change rarement.
- Sur `apps/web` : `handleX` pour les fonctions définies, `onX` réservé aux noms de props (règle Kyle, appliquée sur TopBanner).
- Un snapshot cart **doit être bilingue** (en + fr) : règle métier validée par Rémy. Le schéma `cartProductSchema` est donc la spec ; on corrige les fixtures, pas le hook ni le schéma.
- Factory unique réaliste (`createPublishedProduct` bilingue + `legacyId`), suppression du doublon `createBilingualPublishedProduct`.
- Test « log and skip » : vérifier que le snapshot existant est **préservé** sur recompute en échec (`toEqual(before.cartProduct)`), pas `toBeFalsy`.
