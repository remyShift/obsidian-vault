## Découvertes
- Read localisé : l'adapter lit en `lean: true` (pas de cast mongoose) et le hoisting `afterRead` exige `typeof value === 'object'` → un scalaire plat legacy est servi tel quel par `payload.find`.
- Write localisé : `mergeLocaleActions` reconstruit l'objet entier — locale écrite = valeur entrante, autres locales = `stored[field][locale]` → sur une string plate, `undefined` → le **premier** write sur un doc legacy doit porter la valeur d'origine dans la locale par défaut. Une locale par `payload.update`, pas de write multi-locale.
- Verdict empirique (3 cas) : read scalaire ✓ ; fr-first = perte de données ; en-first puis fr = les deux locales intactes.
- Resolvers translator au runtime dans `payload.config.custom.translator.resolvers` ; `openAIResolver().key === 'openai'` ; `resolver.resolve({ localeFrom, localeTo, req, texts })` appelable depuis une migration, chunke à 100, renvoie `translatedTexts` index-alignés.
- Fallback : `fallback: true` + `defaultLocale: 'en'` → read `fr` vide sert `en` (aussi sur relations peuplées). Déclencheur = null/absent, pas `""`. `fallbackLocale: 'none'` désactive ; `locale: 'all'` renvoie `{ en, fr }` brut.
- Un champ `text` localisé reste typé `string` dans les types générés.
