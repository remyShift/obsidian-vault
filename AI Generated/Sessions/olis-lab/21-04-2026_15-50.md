## Découvertes
- Deux options techniques réelles pour la migration Merchant API : Option A (push 1:1, 2-3 semaines) vs Option B (pivot XML feed + DataSources, 3-4 semaines mais résout GMC + Meta + Klaviyo d'un coup).
- Sub-API **DataSources** de la nouvelle Merchant API permet d'enregistrer un feed XML hébergé (S3/CloudFront) que Google pull automatiquement. Standard de l'industrie, découplage total de la CMS des 3rd parties.
- Doc publique Google **superficielle** sur les détails opérationnels DataSources (formats, schedule, auth feed, limites). Un POC sandbox est indispensable avant engagement sur Option B.
- Breaking changes Merchant API : format ID produit `online:en:GB:SKU` → `en~GB~SKU` (tildes, pas de channel), prix `{value, currency}` → `{amountMicros, currencyCode}`, endpoint `products.update` → `productInputs.patch`, Developer Registration obligatoire (link GCP).
- `pageSize` passe de 250 à 1000 dans Merchant API → 4x moins d'appels pour `list`.
- `@googleapis/merchantapi@17.3.0` est le package le plus ergonomique pour une migration progressive (reproduit le pattern `googleapis` actuel). `@google-shopping/*` en v0.8.0 = pre-1.0, risque d'instabilité.
- Test accounts Merchant API : isolation totale (produits jamais publiés sur Google Search/Shopping), offres auto-approuvées (pas de homepage claiming), quotas identiques à prod, **"Cannot perform registration for test accounts"** = potentiellement bloquant pour tester le Developer Registration E2E.
- Intersection structurelle entre les deux workstreams : si Option B retenue pour le catalogue, le stock doit vivre en DB avant la génération du feed → Bigblue devient prérequis bloquant.
- 4 mois seulement entre aujourd'hui et le sunset Content API (18 août 2026) : timeline serrée, nécessite de commencer rapidement.
