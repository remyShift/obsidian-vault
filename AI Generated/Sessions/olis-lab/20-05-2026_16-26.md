## Decouvertes
- La configuration `defaults: '2025-05-24'` signifie "utiliser le modele de consentement explicite introduit par PostHog le 24 mai 2025" — en v1.335.2 c'etait lache, en v1.369.2 c'est strict
- L'AnalyticsService (packages/shared) est un singleton statique qui wrappe PostHog + GA4 + Meta Pixel — le fix sur `updateConsent()` se propage automatiquement aux 3 outils
- Le hook `web_client/useCookieConsent.ts` importe un fichier local (`service/analytics/AnalyticsService.ts`) qui re-exporte simplement depuis `@olis-lab/shared/analytics` — ce n'est pas un shadowing problematique
- La logique de persistance PostHog est entierement a la charge de l'app (PostHog ne lit plus son propre cookie d'opt-in dans cette configuration)
- Le `posthog.init()` dans `index.tsx` est conditionne par `Object.keys(envOverrides).length > 0` — fragile mais actuellement safe pour tous les envs
- `shopProductSchema.getSearchProducts` et `Bundles.getSearchBundles` ajoutent déjà `type: 'product'` / `type: 'bundle'` sur chaque item retourné — pas besoin de l'ajouter dans le controller
- `isBundle` n'est pas dans le `select` MongoDB de `getSearchBundles` → il faut le dériver de `type` côté frontend
- Pas d'ErrorBoundary sur SearchPage : un crash JS montre une page blanche, pas un état "no results"
- `apiConfig.ts` utilise jotai `getDefaultStore()` mais uniquement dans `handle401Error()` — pas d'effet de bord au chargement du module
- **Payload blocks** : champ de type `blocks` = union typée stockée en tableau JSON, réorderable drag-and-drop dans l'admin, parfait pour page builder
- **Payload Globals** : singleton (une seule entrée, pas une collection) — adapté pour homepage, shop page, footer. Actuellement `globals: []` dans `payload.config.ts`
- Shop page hardcode "Grown Alchemist" en dur dans `Shop.tsx` — cas d'usage direct pour les blocks
- `apps/web` a déjà `PayloadSDK` server-side dans `lib/api/cms.ts` — prêt à consommer des globals avec un simple `getGlobal()`
- **Tests** : `as unknown as T` est inévitable pour les mocks partiels de types complexes — sauf si on extrait la logique en fonction pure avec injection de dépendance explicite
- **Tests** : `data` dans le contexte Payload validator est typé trop large pour être utilisé directement — le cast est à la frontière Payload uniquement, dans le wrapper `uniquePerCategory`
- **Tests** : `vi.fn().mockResolvedValue(...)` satisfait un type `FindFn` minimal sans cast si le type est correctement dimensionné

## Decisions prises
- Ne pas modifier `defaults: '2025-05-24'` ni `cookieless_mode: 'on_reject'` : les deux sont RGPD-corrects, le bug est ailleurs
- Fix minimaliste : ajouter un `useEffect` de montage dans chaque hook pour reappliquer le consentement stocke
- Pour `web_client` : `useEffect(() => {...}, [])` (consent synchrone)
- Pour `apps/web` : `useEffect` avec `useRef` pour detecter le premier chargement du consent (asynchrone a cause du SSR)
- SearchController : pas de fichier `SearchApi.ts` dédié — l'appel axios se fait directement dans le hook (import `api` depuis `apiConfig`)
- `isBundle` derivé du champ `type` avant passage à `mapMixedProducts` — réutilise le mapper existant sans duplication
- Algorithme d'entrelacement : 1 bundle toutes les 2 produits, chaque tableau dans son ordre de pertinence ; les bundles restants s'ajoutent en fin de liste si moins de produits
- **Blocks** : cibler `apps/web` (Next.js) uniquement, pas `web_client` ; shop page migrée avant homepage (scope plus petit, ROI immédiat)
- **Blocks** : 5 block types prévus — HeroBlock, ProductGridBlock, BrandOfMonthBlock, BannerBlock, FeaturesBlock
- **Tests** : pas de tests d'intégration pour `uniquePerCategory` — c'est un validator (pas un hook lifecycle), la logique est couverte en pur unitaire
- **Tests** : extraction de la logique dans `checkUniquePerCategory` avec `find` en paramètre — injection de dépendance explicite plutôt que mock de contexte
- **Tests** : `uniquePerCategory` reste le wrapper Payload — les deux casts (`data`, `req.payload.find`) sont centralisés là, nulle part ailleurs
