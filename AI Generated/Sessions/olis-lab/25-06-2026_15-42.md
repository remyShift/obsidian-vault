## Problèmes / Bugs
- **`Promise cast` sur `data.slug` au create** (erreur initiale) : c'était un **build conteneur périmé** (Dockerfile standalone, chemin `/app`) compilé depuis le commit cassé `c1b516f18` (override supprimé), pas le code corrigé. Le source était bon, l'artefact déployé non.
- **`400 "Slug invalid"` au create publié** (vrai bug, vu après `devsafe`). **Cause racine** : Payload valide les champs **en parallèle** (`Promise.all` dans le `traverseFields` beforeChange). Le slug est posé par le hook **async** de la checkbox (champ frère, `await findByID`), mais la validation `required` du champ slug court en parallèle et lit un slug vide → 400. Le slugField natif s'en sort car son slugify est **synchrone**. → un hook de **champ** async ne peut PAS générer ce slug de façon fiable.
- **Angle mort des tests** : `createPublishedProduct` fournissait un slug, `createDraftProduct` est en draft (validation `required` sautée) → la génération-au-create n'était jamais exercée. Les "20/20" antérieurs étaient un faux filet.
- **CI rouge** (`cms-int-tests`) après merge `develop` : mon test créait deux brands `'Gallinée'`, or `Brand.name` est `unique: true` → collision (`Value must be unique`). Passait en local car l'index unique Mongo n'était pas appliqué (DB fraîche), appliqué en CI.
- **Divergence labels desktop/mobile (PR #1839)** : `resolveNavLabel` était une closure inline dans `Navbar.tsx`, donc inaccessible à `NavbarMenu` (mobile) qui tapait `t(item.labelKey)` en dur. Quand un éditeur CMS renomme un onglet, desktop suivait, mobile non. Cause racine = logique de résolution non partagée.

## Découvertes
- Payload `beforeChange` traverse les champs en **parallèle** (`Promise.all` dans `traverseFields.js`) ; par champ : hooks `beforeChange` **puis** `validate` (`promise.js`). Un générateur de champ async perd la course contre la validation `required` d'un champ frère.
- Ordre opérations Payload : collection `beforeValidate` → field `beforeValidate` → field `beforeChange`+`validate` (parallèle) → collection `beforeChange`. Générer dans **collection `beforeValidate`** = avant toute validation de champ, séquentiel, async OK. C'est l'étage correct (et celui où vit déjà `generateProductSku`).
- Le `/app` dans les stacktraces = build **standalone conteneurisé** (Dockerfile, `server.js`), pas le dev source-mappé → un fix non rebuild/redéployé reste périmé.
- L'index Mongo `unique: true` peut ne pas être appliqué en test local (DB fraîche, index pas construit au moment de l'insert) mais l'est en CI → divergence local/CI réelle.
- `RequiredDataFromCollectionSlug<'products'>` marque `slug` required ; pour tester la génération (slug omis) il faut caster.
- Le bot Devin laisse des commentaires inline sur les PR avec un marqueur de sévérité (🟡 BUG, 🚩 ANALYSIS) ; tri humain nécessaire (ici 1 vrai / 2). Diego répond sur le commentaire à traiter pour signaler la pertinence.
- Les messages next-intl ne sont pas typés strict dans `apps/web` (`labelKey: string` passe dans `t()`) → on peut threader `t: (key: string) => string` dans un helper pur sans souci de contravariance.
