### Découvertes

- Sur **MongoDB**, les migrations Payload ne tracent PAS le schéma (schemaless) ; elles ne servent qu'au backfill data. Pas d'auto-run au déploiement (manuel).
- `payload.db.updateOne/updateMany` (couche adaptateur) écrit le doc principal **sans** toucher les versions ni déclencher les hooks/validation — insuffisant quand l'admin lit la version.
- Avec `drafts: true`, le champ relation lit la version `latest` → il faut peupler la version (donc `payload.update`, qui crée du churn de versions).
- En **Local API**, l'option `locale` n'est pas propagée à `req.locale` (toujours `en`) ; seul le REST de l'admin le positionne. Tests de la locale → appel direct du hook avec `req` contrôlé.
- Importer une collection (`Products`) en env vitest **unit** échoue (chaîne `payload.config` → translator ESM + validation env) → les tests touchant la config doivent être en intégration.
- Mongo standalone (cms_local) = pas de transaction → try/catch+continue OK. Replica set/Atlas (prod) = transaction → une erreur attrapée abort tout.
- `useActiveFeatureFlags()` (posthog-js/react) renvoie `undefined` tant que PostHog n'a pas résolu → c'est le signal pour distinguer le 3e état (loading) du flag off.
- PostHog cache les flags résolus en localStorage → le flash ne touche que le **premier** chargement (fenêtre privée / cache vidé pour reproduire).
- L'idée de Diego (" tester false au lieu de true ") pointait le bon symptôme mais ne corrige pas : un booléen ne peut pas distinguer loading de off.

- Sur erreur CMS, `isReady` passe à `true` quand même → fallback i18n, jamais de loader infini.
