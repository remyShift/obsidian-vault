## Problèmes / Bugs
- `@faker-js/faker` non installé malgré sa présence dans `package.json` — résolu avec `pnpm install` depuis la racine
- `poolOptions` supprimé dans Vitest 4, format déprécié → migré vers `forks.singleFork` puis `threads.singleThread`
- Types Payload : collections avec `versions: { drafts: true }` exigent l'union `draft: true | false` — `draft: false` seul ne suffit pas, `slug` doit aussi être fourni explicitement
- Message d'erreur Payload (`"Content > Slug"`) wraps le message du validateur — assertion sur le message du framework, pas sur notre code
- Race condition dans `initPayload()` : les deux `beforeAll` setup démarrent en parallèle, les deux voient `cached = null`, double initialisation Payload

## Découvertes
- Vitest 4 a supprimé `poolOptions` — les options sont maintenant directement sous `test.forks.*` ou `test.threads.*`
- `pool: 'forks'` + `isolate: false` ne partage PAS le module cache entre fichiers (chaque fork = process séparé, la mémoire ne se partage pas)
- `pool: 'threads'` + `singleThread + isolate: false` améliore la performance (~30% plus rapide) mais Payload s'initialise quand même deux fois à cause de la race condition dans `initPayload()`
- La vraie cause de la double initialisation : les deux `beforeAll` de `vitest.setup.ts` démarrent en parallèle avant que `cached` soit set — classic race condition, fix = cacher la Promise et non le résultat
- Payload Local API n'exige pas l'existence réelle du document référencé pour les champs `upload` — un ObjectId fictif (`'000000000000000000000001'`) est accepté
- `uniquePerCategory` est une `TextFieldValidation` : testable en unit avec `vi.fn()` sur `req.payload.find`, sans DB ni Payload init
- Nos tests d'intégration testent en partie le framework Payload lui-même (propagation d'un string en ValidationError) plutôt que notre logique métier
