## Problèmes / Bugs
- Accès non sûr révélé par le typage strict : `product.translations?.[currentLanguage]?.title.includes(...)` cassait quand `ProductsResult` (`translations: any`) est devenu `ShopProduct` (`translations: TTranslations`). Corrigé en `title?.includes(...)`.
- **`isPopulated is not a function`** au runtime (int tests rouges) + erreurs de types `.category`/`.name` : `dist` de `@olis-lab/shared` périmé (src avait `isPopulated`, pas le dist que cms consomme). Résolu par `pnpm --filter @olis-lab/shared build`.
- Recherche du picker toujours potentiellement lente : vrai coupable = le `like` (regex non ancrée) de Payload qui scanne toute la collection ; raccourcir le label n'aide qu'à la marge.
- `validate` d'un champ Payload : `siblingData` implicite `any` en strict TS → typer inline l'objet d'options (pattern `requiredUnlessBrand` dans `Products.ts`).
- Piège shell : `rm` aliasé en `rm -i` → un `rm -f x && echo ok` affiche le prompt mais ne supprime pas (réponse vide = non). Utiliser `/bin/rm -f` pour forcer.

## Découvertes
- Le flag `dev_search_page` gérait 3 choses : navigation search (HomepageNavbar), route `/search` (Routes), définition (useFeatureFlags).
- Piège : l'ancienne `Search` était utilisée à 2 endroits — HomepageNavbar (gated par flag) ET `Menu.tsx` (menu hamburger mobile, JAMAIS gated). `Menu.tsx` n'était rendu que par `FeedPageNavbar`, lui-même code mort (zéro référence).
- `ProductsResult` (défini dans `expandProducts.tsx`) n'était plus réellement utilisé : le call site de `QuickAddToBagPopup` passe `currentProductAtom` typé `ProductsMinimumInterface`, assignable à `ShopProduct` seul. `CollectionPage` n'utilisait qu'une variable nommée `productsResult` (faux positif), pas le type.
- Le flag `dev_search_page` existe toujours côté PostHog dashboard : le retirer du code ne le supprime pas du dashboard.
- Le workspace cms consomme le **`dist`** de `packages/shared` (pas le `src`) → toute modif de shared exige un `build` avant que cms (runtime ET typecheck via `.d.ts`) la voie. Piège récurrent.
- La recherche d'un `relationship` Payload sur `useAsTitle` utilise un `like` → scan complet, l'index `adminLabel` n'aide pas un regex non ancré.
- Les types générés par Payload pour un array de segments ne sont **pas** une union discriminée : un segment `link` voit quand même `url` comme optionnel/nullable. Le narrowing se fait au rendu via l'asserter `createAsserter` (`packages/shared/src/payload/guard.ts`), pas besoin de Zod.
- Payload n'a pas de « label de ligne par nom de champ » pour les arrays : pour un aperçu lisible dans la liste admin, il faut un composant `RowLabel` (sinon « Message 01 »). `useRowLabel<T>()` donne `{ data, rowNumber }`.
- Ajout d'un composant admin référencé par chemin (`@/components/...`) → il faut régénérer l'importMap (`generate:importmap`) sinon l'admin ne le résout pas.
- Le commentaire en tête de `tradingPlanSchema.ts` indique que le Zod legacy a vocation à être remplacé par l'asserter `guard.ts` → ne pas réintroduire de nouveaux schémas Zod de lecture.
- Navbar : les 3 onglets top-level (`shop`/`your-lab`/`learn`) + `brands` sont codés en dur dans l'UI (`TNavItemType`) → cohérent de les figer côté schéma CMS aussi.
- Navbar : certaines colonnes (ex. Skin) ne sont pas des liens en dur aujourd'hui mais dérivées d'un fetch de TOUS les produits pour en extraire catégories/sous-catégories → fragile, c'est le point « pas propre » que Rémy veut virer.
- Échange Slack/task Michele : la base structure de la navbar (onglets top-level) éditable = pas pour maintenant ; règle posée « si ça ne nous retarde pas on le fait, sinon non » → comme ça retarde, on ne le fait pas. (Commentaire posté par erreur sur la task announcement-bar.)
