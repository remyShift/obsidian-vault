## Découvertes
- **PLP = Server Component dynamique** (`await searchParams`) → hard refresh ET changement de filtre déclenchent une requête serveur → `cms.request('/products-curated')`. **Rien n'est caché** (route dynamique + fetch SDK sans `revalidate`). 1 reload = 1 vague de logs de validation.
- **`assertProduct` n'est pas qu'un validateur, c'est le MAPPER** raw `Product` → `TProduct` (mainImage.url garanti, tags aplatis, cartProduct parsé). Non supprimable côté web : `ProductCard` fait `product.mainImage.url`.
- `assertProduct` valide le produit **entier (PDP)** alors que la PLP ne select/consomme que les champs carte → over-strict. MAIS `cartProduct` **est** requis par la carte (`addToCart(product.cartProduct)` dans `ProductGrid`) → un schéma carte dédié ne sauverait PAS le cas `cartProduct`. C'est un problème de **données**.
- SDK `cms.request({ args })` sérialise via `buildSearchParams` (qs-esm en interne) → pas besoin de `qs-esm` en dép directe d'`apps/web`.
- Global Payload publié pas seedable facilement en test (banners `required` + uploads media) → `resolveCurationSection`/`CuratedQuerySchema` restent en unitaire (frontière comportementale chère).
- `Response.json()` renvoie `Promise<any>` → sans annotation, `data` est `any` (types ne traversent pas HTTP). `z.custom<T>()` = typer sans re-valider au runtime.
- `undefined` locale → Payload fallback `defaultLocale: 'en'` (config `localization`, `fallback: true`), pas de crash.
