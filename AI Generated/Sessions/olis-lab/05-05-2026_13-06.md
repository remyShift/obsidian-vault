### Problemes / Bugs

- Aucun bug rencontre sur le checkout. Session exclusivement dediee a la planification/redaction de user stories.
- `@payload-enchants/seo@1.2.2` incompatible avec Payload `3.79.1` : compile pour `3.0.0-beta.61`, `useFieldProps` supprime de `@payloadcms/ui` stable
- `fields` dans `@payload-enchants/seo` est un tableau statique (AJOUTE des champs), pas une fonction de filtrage ‚Äî impossible de masquer le champ `preview`
- Chemin Playwright dans le plan original incorrect : `/checkout-guest` au lieu de `/secure-checkout`
- Endpoint `searchBrands` initialement place dans le mauvais controller (`products.controller.js` au lieu de `notionBrandsController.js`) ‚Äî corrige en fin de session

### Decouvertes

- Le checkout est dans `apps/web/app/secure-checkout/` et non `checkout-guest/` comme mentionne dans le plan original
- `computePriceBreakdown()` accepte un `voucherDiscountValue` (pourcentage) qui reduit `discountedSubtotalTTC` mais ne touche pas au seuil de livraison gratuite (base sur le sous-total brut)
- La delivery step a une logique `canContinue()` qui bloque si relay selectionne sans pickup point - non couverte dans le plan initial
- `createStepReducer` et `canContinue` probablement pas encore exportes - a verifier avant d'ecrire les tests
- Distinction importante : les tests E2E doivent valider que le flow complet aboutit (order created), pas que des elements HTML s'affichent
- `@payload-enchants/seo` exporte `seo` (pas `seoPlugin`), son option `fields` AJOUTE des champs au lieu de les filtrer, et ses types ne sont pas exposes dans l'exports map (inaccessibles avec `moduleResolution: bundler`)
- `useFieldPath` (Payload stable) retourne directement une `string`, contrairement a `useFieldProps` (beta) qui retournait `{ path: string }`
- L'approche native du plugin officiel (`generateTitle`/`generateDescription`) suffit pour brancher OpenAI ‚Äî zero infra custom necessaire
- `brandsModel.searchByBrand` existait deja dans le serveur (Fuse.js, threshold 0.2) mais n'etait connecte a aucun endpoint ‚Äî reutilise as-is
- L'encoding des noms de marques avec `/` utilise `---` comme separateur (ex: "Mid/night's" ‚Üí `Mid---night-s`) ‚Äî reproduit dans le `Link` du composant
- Le `notionBrandsRouter` est monte sur `/brands` dans `server.js` (pas `/products`) ‚Äî toute logique brands doit aller dans ce router
- [Search] `validateInput.js` existe mais est vide ‚Äî prevu comme point d'entree partagee pour la validation, jamais implemente
- [Search] Les 5 endpoints search ont des conventions de nommage incoherentes (`search-products`, `search-actives`, `search-articles` vs juste `search`)
- [Search] Brands : aucun guard `if (!query)` du tout ‚Äî le seul endpoint sans validation manuelle
- [Search] La SearchPage fait 5 requetes HTTP paralleles par frappe ‚Äî bornee par la plus lente (appels Notion souvent plus lents que Mongoose)

### Decisions prises

- User stories ecrites au niveau metier (pas de variables de code, pas de valeurs hardcodees)
- Pour les schemas : on teste uniquement ce qu'on a ecrit (refine custom, cross-field billing), pas la validation native Zod
- Order payload builder exclu du perimetre de tests unitaires pour l'instant (backend concern)
- Tests E2E : focus sur le flow (l'ordre est cree, le montant est correct), pas sur l'UI (tel element est visible)
- Granularite claire : validation/gating/calculs dans les unit tests, happy path + recovery dans les E2E
- SEO : rester sur `@payloadcms/plugin-seo` officiel avec `generateTitle`/`generateDescription` branchees sur OpenAI ‚Äî pas de plugin tiers, pas de patch
- Brands search : liste de liens (pas de cards) ‚Äî design intentionnellement minimal, coherent avec la demande
- Brands query sans `lang` (les noms de marques ne sont pas localises)
- `searchBrands` appartient a `notionBrandsController` / `notionBrandsRouter` ‚Äî coherence avec l'architecture existante du serveur
- [Search] Option A recommandee (endpoint unifie `GET /search?q=&lang=`) : 1 requete au lieu de 5, validation en un seul endroit, controller SRP
- [Search] Ne pas toucher aux anciens endpoints ‚Äî backward compat pour la navbar (`/products/search-suggestions`)
- [Search] Pas stocker la query brute en analytics (GDPR) ‚Äî tracker uniquement les counts de resultats par categorie
