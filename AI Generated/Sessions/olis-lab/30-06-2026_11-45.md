## Découvertes
- PLP Next = fonction pure `URL → HTML` : l'URL est la seule source de vérité (filtres/tri/limit), serveur = réducteur, `optimisticFilters` côté client = cache d'affichage qui converge via `router.replace`. Pas de React Query, pas de Jotai pour les filtres.
- Pagination PLP Next = "Show More" par **accumulation de `limit`** (+20), `page` toujours = 1 → refetch from scratch à chaque clic, `depth:2` + `assertProduct` sur tout → O(n²) en données/CPU si le catalogue grossit.
- `buildFilterUrl` supprime `limit` à chaque changement de filtre/tri → la pagination se reset, comportement voulu et correct.
- `buildWhereClause` force toujours `status:'Live'` ET `_status:'published'` (double statut produit).
- Curation legacy = **config-based**, pas de champ produit : collection Mongo `curation_settings` (`brandOfTheMonth`, `prioritySkus`, `maxPerBrand`), facets MongoDB + `interleave()` round-robin.
- **Curation déjà migrée dans Payload** : global `trading-plan` (`apps/cms/src/globals/TradingPlan.ts`) a `curationSettings.{homepage,shop}` avec `brandsOfTheMonth` (relation → brands, **ordre signifiant**), `prioritySkus` (relation → products), `maxPerBrand`. Type `TradingPlan` (`payload-types.ts:1343`), guard `assertTradingPlan`.
- Le SDK `@payloadcms/sdk` expose `.request({ method, path, args })` pour taper un endpoint custom (`/api/...`).
- Endpoints custom Payload : 4 existants (tous POST + auth) dans `apps/cms/src/endpoints/sync/*`, enregistrés dans `payload.config.ts` (array `endpoints`). Collection `products` a `access.read: () => true` → un endpoint **public read** peut omettre la garde `if (!req.user)`.

## Décisions prises
