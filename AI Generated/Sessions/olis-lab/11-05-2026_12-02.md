# Recap — olis-lab — 11/05/2026

## Avancées
- Plan d'automatisation des feeds XML (GMC + Klaviyo) rédigé et documenté en anglais selon le template équipe
- 4 options analysées avec pros/cons détaillés : Payload afterOperation hook, admin button, CRON node-cron, AWS EventBridge
- Recommandation claire formulée : Option C (CRON) comme primaire + Option B (admin button) comme fallback manuel

## Problèmes / Bugs
- Bug S3 double déclaration XML toujours ouvert (non adressé cette session)

## Découvertes
- Un Payload `afterOperation` hook sur les produits déclencherait un appel BigBlue complet à chaque save — risqué en cas d'import bulk ou d'éditions répétées
- La fraîcheur maximale nécessaire pour GMC est ~24h (Google pull le feed une fois par jour) — un CRON horaire est largement suffisant
- EventBridge n'est justifié que si le server devient stateless (Fargate/Lambda) — overkill pour l'architecture actuelle
- Le bug S3 double déclaration XML et l'absence d'auth sur `/sitemap/*` sont des prérequis à corriger avant d'automatiser : automatiser un pipeline cassé ne fait que répéter les erreurs en boucle

## Décisions prises
- Stratégie retenue : **CRON (node-cron) + admin button Payload** — pas de hook afterOperation pour l'instant
- Les triggers automatisés devront utiliser un shared secret header sur `/sitemap/*` avant mise en prod (auth tech debt identifié comme prérequis)
- L'Option A (hook Payload) reste une évolution possible si la fraîcheur temps-réel devient un besoin business

## Prochaines étapes
- Résoudre le bug S3 double déclaration XML (bloquant avant toute automatisation)
- Ajouter un shared secret header sur les endpoints feed (prérequis sécurité avant CRON)
- Implémenter le CRON `node-cron` dans le server Express pour les deux feeds
- Ajouter le bouton admin Payload "Regenerate feeds" comme fallback manuel
- Implémenter le feed Meta (`feeds/meta-${lang}.xml`) — RSS 2.0, `in stock` avec espaces
- Valider les feeds GMC + Klaviyo dans leurs sandboxes respectifs
