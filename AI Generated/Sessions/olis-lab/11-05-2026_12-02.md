## Découvertes
- Un Payload `afterOperation` hook sur les produits déclencherait un appel BigBlue complet à chaque save — risqué en cas d'import bulk ou d'éditions répétées
- La fraîcheur maximale nécessaire pour GMC est ~24h (Google pull le feed une fois par jour) — un CRON horaire est largement suffisant
- EventBridge n'est justifié que si le server devient stateless (Fargate/Lambda) — overkill pour l'architecture actuelle
- Le bug S3 double déclaration XML et l'absence d'auth sur `/sitemap/*` sont des prérequis à corriger avant d'automatiser
- Le plugin `@payload-enchants/translator` fournit le bouton "Translate" out of the box — aucun hook custom nécessaire pour le cas d'usage manuel
- Pour automatiser la traduction au save il faudrait un `afterChange` hook custom appelant `translateOperation()` — non retenu
- `apps/cms/src/sync/` contient des schemas Zod + un transformer Payload → Legacy (sync-back) mais pas de script Legacy → Payload
- La connexion à la legacy DB est déjà configurée (`LEGACY_DB_URI` dans Zod, `src/lib/db/legacy.ts`)
- Masquer un bouton plugin Payload par locale nécessite un composant React provider + `useLocale()` + inspection DOM pour trouver le sélecteur CSS — complexité disproportionnée

## Décisions prises
- Stratégie feeds retenue : **CRON (node-cron) + admin button Payload** — pas de hook afterOperation pour l'instant
- Les triggers automatisés devront utiliser un shared secret header sur `/sitemap/*` avant mise en prod
- L'Option A (hook Payload) reste une évolution possible si la fraîcheur temps-réel devient un besoin business
- Approche B retenue pour les traductions (plugin vs hook custom) : moins de code, bouton UI inclus
- `OPENAI_API_KEY` en required dans Zod (fail fast si absente)
- Modèle `gpt-4o-mini` (plus récent et moins cher que `gpt-3.5-turbo` déprécié)
- Workflow traduction : manuel via bouton, Suze relit — pas d'automatisation au save pour l'instant
- Masquage du bouton "Translate" sur locale EN : **abandonné**, trop complexe pour le gain
