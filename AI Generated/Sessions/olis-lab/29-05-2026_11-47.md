- **Root cause 2** : Payload retourne un groupe vide `{}` pour `howToUse` sur les produits Collectibles (pas `null`/`undefined`), ce qui faisait passer le guard et crashait sur `assert.required(howToUse.instructions)`
- Le linter a modifie `cartProduct.ts` en session (rend `en`/`fr` requis sans `.default({})`) — cause 33 echecs supplementaires detectes immediatement

## Decouvertes
- `z.unknown()` dans un `z.object()` Zod est **requis** par defaut (pas optional) — necessite `.optional()` explicite
- `minRows` sur un array Payload n'equivaut pas a `required: true` — les deux sont necessaires
- La divergence `brandsOfTheMonth` vs `brandOfTheMonth` n'etait pas detectee par TypeScript (structural typing)
- Payload retourne les groupes non remplis comme `{}` (pas `null`/`undefined`) — les guards qui testent `!= null` ne suffisent pas
- Le `dist/` de `packages/shared` peut devenir stale si on ne rebuild pas apres chaque changement source — source de bugs silencieux en production (Next.js app importe depuis dist)
- `title`/`description` du `cartProduct` legacy sont dans `translations.fr.title` / `translations.en.title`, jamais a la racine du document MongoDB
- `@payloadcms/sdk` fonctionne dans un CRA (HTTP via fetch, pas de dependance Node server) — le package `payload` lui ne marcherait pas cote browser
- Precedence operateurs : `flag && cms?.x ?? fallback` = `(flag && x) ?? fallback` ; flag `false` -> `false`, et `false ?? fallback` reste `false` (le `??` ne fallback que sur null/undefined). Toujours gater via ternaire a la source
- `typeof image === 'string'` dans une reponse Payload = ID non peuple (pas une URL) -> doit retourner `''`
- Pattern `var(--bg-img, url('fallback'))` en SCSS + set inline conditionnel = image dynamique CMS avec fallback asset local propre

- Throw plutot que warn pour les refs non peuplees dans l'asserter

- `cartProductSchema.translations.en` et `.fr` sont **requis** (les deux langues doivent etre presentes dans le JSON synce)
- `title`/`description` supprimés de la racine de `cartProductSchema` (structure reelle des données Payload)
- Feature flag CRA via PostHog (`useFeatureFlags` / `dev_payload_trading_plan`), pas via env var
- Tous les types de props du trading plan derivent du schema Zod (`TTradingPlanViewData`), zero redeclaration manuelle
- Image CMS cablee partout via CSS var avec fallback asset local
- `tradingPlanSchema.ts` marque comme temporaire (a remplacer potentiellement par l'asserter)
- Gating du flag fait a la source par ternaire (types props intacts), flag visible explicitement dans les pages
