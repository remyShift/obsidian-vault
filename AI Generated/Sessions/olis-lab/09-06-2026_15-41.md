## Découvertes
- **Mongoose pluralise le nom de collection** : le slug `product-media` est stocké dans la collection Mongo `product-medias` (idem `logo-media`→`logo-medias`). Un insert/delete DB brut doit viser le nom pluralisé (sinon `findByID` → NotFound).
- `initPayload()` (`tests/helpers/payload.ts`) est déjà un **singleton mémoïsé** → `beforeAll(initPayload)` ne boote Payload qu'une fois par fichier ; c'est déjà optimal (Vitest isole les modules par fichier, workers = process séparés). Le coût par test n'est pas l'init mais la **création des fixtures** (~6 écritures DB/test).
- Dans un hook `beforeChange`, le `data` reçu est le **doc fusionné** (pas juste le patch) : `data.mainImage`/`data.brand` sont présents même quand l'update ne les touche pas → le hook fait les `findByID` correspondants.
- L'ordre Payload : la validation des fields tourne **avant** `beforeChange`. Donc un `validate` sur le field `cartProduct` ne voit jamais la valeur calculée par le hook → la vraie garde runtime est le `parse()` dans le hook.
- Une brand créée en `draft: true` reste résolvable par `findByID`/populate (le doc principal porte le contenu draft) → permet de sauter les uploads requis (logo) tout en gardant la relation utilisable.
- Pour faire throw `findByID` en intégration (US-8) : un **brand id inexistant** (`'0'.repeat(24)`) suffit — Payload ne valide pas l'existence d'une relation.
- `payload.update` accepte un `context` → permet de tester `skipCartSnapshot` en intégration.
- **Tentative `typescriptSchema` (json field) avortée** : `z.toJSONSchema` (zod 4) ne marche pas sur `cartProductSchema` (shared est en **zod 3.25.76**, cms en **zod 4.3.6**) ; bumper shared seul recréerait le mismatch chez `web`/`web_client`/`api` (tous zod 3). Personne ne lit le type généré brut `Product['cartProduct']` (la forme typée circule via `TCartProduct`/`assertProduct`) → typescriptSchema abandonné.
- Un relationship picker Payload n'affiche/cherche que sur **un seul** champ (`useAsTitle`). Pas de recherche multi-champs native → un champ label dénormalisé est la réponse idiomatique.
- Modèle actuel : la relation vit sur `Edits.products` (`hasMany`), donc on assigne **depuis le document Edit**. Aucun champ sur Product ne pointe vers Edits.
- Bulk Edit natif de Payload ne modifie que les champs **de la collection listée** → impossible d'assigner des produits à un Edit depuis la liste Products en l'état.
- Plugin Nested Docs = hiérarchie parent/enfant **dans une seule collection** (champ `parent` auto-référencé + `breadcrumbs` auto-calculés). Orthogonal au problème de recherche, et sans effet sur le label produit (`title | brand | sku`, un produit n'étant pas une hiérarchie).
- **L'union `string | T` des relations est dans les types générés, et n'est narrow ni par le SDK ni par la Local API, quelle que soit la `depth`.** Limitation universelle Payload (d'où le PR communautaire ouvert). La `depth` est un fait runtime que les types ne capturent pas.
- La **Local API** (`getPayload({config})` → `payload.find()`) ne marche qu'**in-process** dans l'app qui possède `payload.config.ts` (server components / API routes / hooks, jamais un navigateur). Elle supprime la frontière HTTP, pas le typage de l'union.
- Ce que les exemples Payload évitent n'est pas l'union mais le **besoin d'une couche asserter partagée à l'échelle** : ils consomment en in-process, co-localisé, et narrow l'union **inline**. Olis Lab a Payload déployé séparément + 2 fronts HTTP → contrat partagé cross-package/réseau → couche asserter réelle nécessaire.
- Comme la donnée traverse le réseau (HTTP), la **validation runtime défensive est plus justifiée** chez Olis Lab que dans un exemple Payload in-process → un wrapper `as`-only est encore moins défendable.
- Le guard `assertProduct(raw: Product)` est déjà typé en entrée → un champ Payload supprimé/renommé casse déjà le build. Le trou = champ **ajouté** ou optionalité changée (drift silencieux).
- **Tous** les slots `collection.admin.components` de la list view (`beforeList`, `Description`, `listMenuItems`, `beforeListTable`, `afterListTable`) sont rendus **à l'intérieur** du `SelectionProvider` → `useSelection()` (qui expose `getSelectedIds()`, `count`) fonctionne dans n'importe lequel.
- **Aucun slot public** dans la barre native "X selected / Edit / Publish / Delete" (`ListSelection`) en Payload 3.84 → s'y greffer imposerait de remplacer toute la list view (`admin.components.views.list`), disproportionné.
- Le bulk-edit natif (`EditMany/DrawerContent.js`) fait un `PATCH` qui **remplace** la valeur du champ et force `_status: "published"` (ou `"draft"`). Confirme : sur un `hasMany`, il écrase le tableau, il ne sait pas append.
- **Le vrai pivot du choix d'approche = append vs replace, pas la direction de la relation.** Dès qu'on exige l'append (ajouter à un Edit sans virer l'existant), on sort du natif → code custom obligatoire quelle que soit la collection qui porte le champ.
- Approche A (champ `Products.edits`) casserait le sync (le transformer lit `Edits.products`) sauf à inverser la source de vérité = refacto plus lourd, et reste en sémantique replace.
