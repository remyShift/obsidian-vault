## Découvertes
- Un relationship picker Payload n'affiche/cherche que sur **un seul** champ (`useAsTitle`). Pas de recherche multi-champs native → un champ label dénormalisé est la réponse idiomatique.
- Modèle actuel : la relation vit sur `Edits.products` (`hasMany`), donc on assigne **depuis le document Edit**. Aucun champ sur Product ne pointe vers Edits.
- Bulk Edit natif de Payload ne modifie que les champs **de la collection listée** → impossible d'assigner des produits à un Edit depuis la liste Products en l'état.
- Plugin Nested Docs = hiérarchie parent/enfant **dans une seule collection** (champ `parent` auto-référencé + `breadcrumbs` auto-calculés). Orthogonal au problème de recherche, et sans effet sur le label produit (`title | brand | sku`, un produit n'étant pas une hiérarchie).
- **L'union `string | T` des relations est dans les types générés, et n'est narrow ni par le SDK ni par la Local API, quelle que soit la `depth`.** Limitation universelle Payload (d'où le PR communautaire ouvert). La `depth` est un fait runtime que les types ne capturent pas.
- La **Local API** (`getPayload({config})` → `payload.find()`) ne marche qu'**in-process** dans l'app qui possède `payload.config.ts` (server components / API routes / hooks, jamais un navigateur). Elle supprime la frontière HTTP, pas le typage de l'union.
- Ce que les exemples Payload évitent n'est pas l'union mais le **besoin d'une couche asserter partagée à l'échelle** : ils consomment en in-process, co-localisé, et narrow l'union **inline**. Olis Lab a Payload déployé séparément + 2 fronts HTTP → contrat partagé cross-package/réseau → couche asserter réelle nécessaire.
- Comme la donnée traverse le réseau (HTTP), la **validation runtime défensive est plus justifiée** chez Olis Lab que dans un exemple Payload in-process → un wrapper `as`-only est encore moins défendable.
- Le guard `assertProduct(raw: Product)` est déjà typé en entrée → un champ Payload supprimé/renommé casse déjà le build. Le trou = champ **ajouté** ou optionalité changée (drift silencieux).
