## Découvertes
- La branche `feat/subcategories-payload-admin-label` était au départ strictement identique à develop (0/0). Toute l'infra (migrations, `isPopulated`, `composeAdminLabel`) ne vit que sur la branche produit non mergée `feat/edits-product-picker-admin-label`.
- Localiser le champ admin label **simplifie** le hook : plus besoin de forcer l'anglais (`LABEL_LOCALE`/`resolveEnglishName` supprimés), le hook `beforeChange` tourne une fois par locale sauvegardée et Payload stocke la valeur sous cette locale.
- Le bon type pour un `validate` Payload sur champ `text` = `string | null | undefined` (pas `string` : la valeur est `null`/`undefined` quand le champ est vide/non saisi, c'est tout l'intérêt du garde `value == null` ; pas `unknown` : trop défensif vu le contrat connu). Après le premier garde, TS narrow → le check `typeof value !== 'string'` devient inutile et saute.
- La validation `link` existante de l'announce-bar ne faisait QUE le required conditionnel, **pas** le format → mutualiser via `isValidLink` ajoute la validation de format sur announce-bar ET trading-plan (changement de comportement assumé, risque data faible : contenu trading plan prod pas encore rempli).
- Pas encore de dossier `fields/` ou `validation/` dans le cms : les helpers de validation vivaient inline dans les collections (`requiredUnlessBrand`, `uniquePerCategory`). `lib/validateLink.ts` inaugure le pattern partagé.
- `req.locale` (= `PayloadRequest['locale']`) vaut `'en' | 'fr' | 'all' | undefined`, plus large que `Config['locale']` (`'en' | 'fr'` seulement) généré dans `payload-types.ts`. Cette version de Payload n'exporte pas `TypedLocale`.
- L'adapter `mongoose` auto-découvre `./src/migrations` : aucune modif de `payload.config.ts` nécessaire pour câbler les migrations.
- `fallback: true` actif sur la localisation : un label FR manquant retombera sur l'EN au calcul.
- **Format de nom de migration Payload** : `payload migrate:create` génère le nom via `createMigration.js` → `new Date().toISOString().split('T')`, puis `${yyyymmdd.replace(/\D/g,'')}_${hhmmss.split('.')[0].replace(/\D/g,'')}_${name}.ts`, soit `YYYYMMDD_HHMMSS_name`. Donc le format n'est pas arbitraire. Payload exécute les migrations dans l'ordre **lexicographique du nom** → `Date.now()` (epoch, commence par `1...`) trierait **avant** les noms en format date (`2026...`), cassant l'ordre si on mélange les deux. Argument décisif pour ne pas remplacer.
- **SKU** — un champ `required` Payload **ne peut pas** être rempli par un hook serveur seul : la validation required tourne **côté client avant l'envoi de la requête**, donc aucun hook serveur (beforeValidate/beforeChange) ne tourne à temps. Solutions : validate custom qui relâche à la création, OU générer côté client (ce que faisait le bouton), OU non-required.
- **SKU** — `required: true` + `validate` custom coexistent : la validate custom **remplace** la validation required par défaut (renvoie `true` à la création → pas de blocage client, exige sku à l'update). Le type généré reste `string` (basé sur `required`, ignore la validate).
- **SKU** — le pattern asserter du codebase ne sert qu'à ce que le type Payload ne peut pas exprimer (population de relation, url media, shape json) — **pas** à se prémunir des drafts vides. `ean`/`title`/`price` sont required et non assertés → `sku` doit suivre la même règle.

- **SKU — bouton supprimé** : pas de preview, pas d'endpoint. Génération uniquement au save (serveur). Edge assumé : un draft sauvé sans EAN devra être re-sauvé en draft (le hook tourne au save draft) avant de pouvoir être publié.
- **SKU — wrapper d'erreur** `assertProduct` conservé (inclut `error.message` + ZodError en `cause`) : comportement attendu = Error uniforme avec contexte produit.
- **Validation lien mutualisée** dans `isValidLink` (lib/), appliquée announce-bar + trading-plan ; la navbar branchera dessus. `#` retiré de la regex par Rémy (placeholders gérés autrement). Type `validate` = `string | null | undefined`.
