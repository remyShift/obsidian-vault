## Découvertes
- En JS `\w` = `[A-Za-z0-9_]` → inclut les chiffres. Le slugify ne supprime PAS les nombres (clarification demandée par Rémy).
- Tous les slugs produits ne dérivent pas de la même locale : 1 seul vient du FR (Misaj), les 23 autres de l'EN. D'où la stratégie locale-source.
- Sync Payload → legacy DB (`shop_products`) est **manuelle** (bouton admin → `POST /api/sync/product/:id`), pas d'`afterChange`. Un `payload.update` ne propage pas tout seul. Brands : pas de slug dans le shape legacy (`brand: brand.en.name`), rien à propager.
- `apps/server/utils/generateSlug.js` (Express legacy) a le même bug mais n'est pas le chemin qui fait foi → hors scope.

## Décisions prises
- Scope = **cms uniquement** (pas le serveur Express legacy).
- Slugify : conserver le comportement sur `/` et `.` (strip, pas remplacement par tiret) → ne change pas les brands non visées (ex. `Mid/night 00.00`).
- Migration = changement minimal préservant la locale source (pas de recompute en bloc qui casserait les URLs figées et nos décisions).
- Misaj et cas similaires (titre disparu) : **warn + re-save manuel admin**, pas de recompute auto en bloc.
- `grown-alchemist-skin-renewal-toner` (slug propre, titre dérivé +Mist) : laissé tel quel, pas un bug.
- `face-body-serum` : fix minimal (pas d'ajout du préfixe brand `irritated-`).
- Pas de tests automatisés pour ce fix (demande de Rémy).
- Propagation legacy = étape séparée via sync existant (pas de nouveau code legacy).
