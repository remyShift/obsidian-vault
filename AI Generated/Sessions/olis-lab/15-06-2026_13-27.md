## Découvertes
- L'`importMap.js` Payload dépend de l'état exact du working tree : il ne réécrit que si un import est NOUVEAU vis-à-vis du fichier existant. Un working tree sale donne un résultat trompeur.
- `req.payload` === singleton `payload` du test → `vi.spyOn(payload.logger, 'error')` capte bien l'appel fait par `reportError(req.payload.logger, …)` dans le hook.
- `generate:types` lance aussi `tsx scripts/sync-payload-types.ts` qui propage vers `packages/shared/src/payload-types.ts` (3e fichier vérifié par la CI).
- Les tests d'intégration CMS tournent sur MongoDB (`adapter.connection.dropDatabase()`) lancé par la CI — non reproductible proprement en local.
- Les 3 jobs CI : `build` (typecheck + verify generated files), `cms-int-tests`, `changes`.
- PostHog est câblé dans `apps/web` (provider + `lib/posthog.ts`) **uniquement pour analytics/error reporting** : aucun hook ni système de feature flag n'existe côté Next. Le `useFeatureFlags` (registry + camelCase + mocks) vit seulement dans `apps/web_client`. Le nouveau hook est le 1er consommateur de FF côté Next.
- `@payloadcms/sdk` expose `findGlobal(options, init?: RequestInit)` ; le 2e arg est passé à `fetch` → on peut y mettre `{next:{revalidate, tags}}` (cache Next idiomatique, plus propre que `unstable_cache`).
- `apps/web` a un `Link` locale-aware dans `@/i18n/navigation` (next-intl), et `@olis-lab/shared/routing` exporte `isExternalLink` + `isNextRoute` (NEXT_JS_ROUTES = `/products`, `/checkout-guest`) — utile car certaines routes internes (`/edits/...`) appartiennent au CRA, pas à Next.
- `apps/web` est en **Tailwind v4** (config CSS-based : `@import 'tailwindcss'`, pas de `tailwind.config.*`). Tailwind ne scanne pas `node_modules` → pour styler un composant importé de `@olis-lab/ui`, il faut un `@source` pointant vers le source du package.
- Le `TopBanner` de `@olis-lab/ui` n'a pas de `'use client'` mais fonctionne monté dans une frontière client ; package pré-buildé (`dist/index.mjs`) → pas besoin de `transpilePackages`. Précédent : `CookieBanner` (apps/web) consomme déjà `useCookieConsent` du package UI.
- Retour de Kyle (lead) : suivre le même process que `web_client` pour lire les flags (`useActiveFeatureFlags`, `LOCAL_DEV_MOCKS`) ; setup SSR dispo si besoin (pas nécessaire ici).
- Embla reste utilisé ailleurs dans apps/web (`components/ui/carousel.tsx`, `Footer/FooterTrustBar.tsx`) → deps conservées.
