## Découvertes
- Payload blocks = union typée en tableau, drag-and-drop dans l'admin, chaque item peut avoir une structure différente (≠ array field où tous les items partagent le même schéma)
- Payload Globals = singleton éditable depuis l'admin, parfait pour homepage/shop page — `globals: []` vide dans le projet
- La distinction clé blocks vs champs groupés : les blocks n'ont de valeur que si les éditeurs ont besoin de réordonner/ajouter/supprimer des sections ; pour éditer le contenu dans une structure fixe, un Global plat suffit
- Le trading plan et le page builder sont deux choses distinctes : trading plan = données métier (quelle brand, quelles catégories), page builder = comment les afficher (layout, ordre des sections)
- Michele confirme que le layout change "sooner than we think" → blocks justifiés à terme
- L'approche backsync Global Payload ↔ legacy DB permet de ne pas toucher à CRA immédiatement tout en centralisant l'édition dans Payload

## Décisions prises
- Trading plan phase 1 : créer un Global Payload en miroir de la collection legacy DB existante avec backsync
- Trading plan phase 2 : étendre les deux collections (Payload Global + legacy DB) avec les champs manquants (textes + images hardcodés)
- Trading plan phase 3 : mettre à jour CRA pour lire depuis la DB plutôt que les valeurs hardcodées

- Pas de connexion CRA directe au Global Payload pour le page builder — la migration Next.js est la bonne cible à terme
