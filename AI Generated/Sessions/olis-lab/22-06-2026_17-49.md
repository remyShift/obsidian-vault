## Découvertes
- Config PostHog client (`apps/web/lib/posthog.ts`) : `cookieless_mode: 'on_reject'`, opt-out par défaut, consentement custom `olislab_consent`, région EU (`ui_host: eu.posthog.com`).
- PostHog recommande désormais la **feature flags secure API key** (`phs_`, **project-specific**, dans Project Settings → onglet Feature Flags) plutôt qu'une personal API key account-wide (`phx_`) pour l'éval locale. Les deux passent par l'option SDK `personalApiKey`.
- Doc PostHog : ne **jamais** inventer d'UUID côté serveur ; pas de cookie → skip serveur, laisser le client trancher (source de notre hybride). Réf : `/docs/integrations/v0`, `/docs/feature-flags/local-evaluation`, `/docs/feature-flags/remote-config`.
- La doc Next.js de PostHog utilise exactement un singleton `lib/posthog-server` / `getPostHogServer()` (même pattern qu'on a construit).
- Éval locale = télécharge les **définitions** de flags (opération privilégiée → clé secrète) ; éval remote = appel `/flags` par requête (facturable + latence TTFB).
- Le champ `app` des nav items apps/web (`'legacy'` = `<a>` full-page routé par CloudFront, `'next'` = `<Link>`, `'placeholder'` = disabled) : choisi `'legacy'` pour les paths CMS internes (seul choix universellement sûr pendant la migration).
- Un server component ne peut pas lire le flag client (`useFeatureFlags` est `'use client'`) → d'où l'instance serveur.

## Décisions prises
- **Architecture** : fetch server-side + gate client (idiomatique apps/web, mirror `getAnnouncementBar`), pas de React Query client.
- **Labels top-level** pilotés par le CMS quand flag ON (parité avec web_client).
- **Pas de tests** pour ce port (choix Rémy ; les transforms restent des fonctions pures testables).
- **Éval serveur locale** (secure key), pas remote ; gestion **hybride** (cookie → serveur, sans cookie → fallback client).
- **Gate strict `=== true`** (edit Rémy) : fetch seulement si le serveur confirme ON. Trade-off : sans cookie/sans clé → legacy ; mitigé par le mock local serveur.
- **Mutualisation** `TFeatureFlag` + `LOCAL_DEV_MOCKS` dans un module neutre (ni `'use client'`, ni server-only).
- **Pas de `flagsReady` serveur** : inutile (éval awaitée par requête) ; `null` encode déjà "indécis → défère au client".
- posthog-node `^4.18.0` (compat Node repo).
