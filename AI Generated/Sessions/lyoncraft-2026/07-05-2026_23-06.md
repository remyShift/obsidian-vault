## Problèmes / Bugs
- `ERR_PNPM_IGNORED_BUILDS` en CI pour esbuild@0.25.12, playwright-chromium@1.59.1, vue-demi@0.14.10 — exit code 1
- `pnpm approve-builds` localement : "no packages awaiting approval" (packages deja dans le store, pas de check declenche)
- `--no-frozen-lockfile` n'a pas resolu le probleme : pnpm v10 bloque quand meme les scripts au niveau du check d'approbation
- Ajout manuel de `settings.onlyBuiltDependencies` dans le lockfile inefficace : pnpm v10 ne lit pas ce champ au bon endroit

## Découvertes
- pnpm v10 bloque tous les build scripts par defaut (`ERR_PNPM_IGNORED_BUILDS`) — nouveau systeme d'approbation strict introduit en v10
- En local, pas d'erreur parce que les packages sont deja dans le store global pnpm : les scripts d'installation ne re-tournent pas → le check d'approbation n'est jamais declenche
- En CI (environnement vierge, pas de store cache), pnpm telecharge tout de zero et doit executer les scripts — c'est la que le check bloque
- `onlyBuiltDependencies` dans `package.json` sous la cle `pnpm` est cense suffire selon les docs, mais ne fonctionne pas en CI avec pnpm v10 (au moins dans les versions recentes)
- Le format lockfile `9.0` est identique entre pnpm v9 et v10 : compatibilite totale, pas de migration necessaire

## Décisions prises
- Pincer pnpm a `version: 9` dans le workflow CI plutot que `latest` — v9 n'a pas le systeme d'approbation strict, les scripts tournent librement
- Garder pnpm v10 en local avec `onlyBuiltDependencies` dans `package.json` pour la securite en dev
- Ne pas conserver `settings.onlyBuiltDependencies` dans le lockfile (ce n'est pas le bon mecanisme)
