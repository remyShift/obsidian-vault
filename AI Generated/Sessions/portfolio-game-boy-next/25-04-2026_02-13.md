## Problèmes / Bugs
- Bug flash boot identifié et corrigé : `useLayoutEffect` post-mount provoquait un cycle render→effect qui rendait l'overlay visible 1 frame au remount client après navigation.
- Premier test Screen passait par accident car `useLayoutEffect` flushe dans `act()`. Renforcement nécessaire avec un spy sur le composant mocké pour capturer le premier render.

## Découvertes
- `useState(initializer)` lit la valeur initiale avant le premier render, ce qui évite le flash que `useLayoutEffect` ne peut pas éviter (l'effect court après commit, donc l'utilisateur peut voir 1 frame avant le re-render).
- Les tests React Testing Library qui interrogent le DOM final masquent les bugs visuels de flash : il faut spy sur le composant lui-même pour vérifier qu'il n'a jamais été monté.
- La CSP de Next.js sur Vercel doit inclure `vercel.live` + `*.vercel-insights.com` + `*.vercel-scripts.com` pour ne pas casser Speed Insights.
- framer-motion injecte des styles inline → CSP `style-src` doit autoriser `'unsafe-inline'`. Pas idéal mais pas de contournement propre sans réécrire toute l'animation côté CSS.
- Toutes les `<Image>` SVG du chrome GameBoy étaient déjà bien marquées `aria-hidden` et `alt=""` — Phase 7.5 plus rapide que prévu.

## Décisions prises
- Focus visibles implémentés en CSS global dans `globals.css` plutôt qu'en classes Tailwind `focus-visible:ring-*` dupliquées sur chaque composant. Avantage : un seul point de modification, pas d'oubli possible. Inconvénient : moins découvrable côté JSX.
- Honeypot et `data-boot-overlay` testés via assertions sur les mocks (Resend non appelé, spy de rendu) plutôt que via DOM. Plus robuste contre les changements de markup.
- CSP avec `'unsafe-inline'` sur `style-src` accepté comme trade-off pour conserver framer-motion. À revisiter si on migre vers des animations CSS pures.
- Rate-limit en mémoire conservé (vs Redis/KV) car volume actuel négligeable. Limite documentée pour ne pas piéger un futur lecteur.
