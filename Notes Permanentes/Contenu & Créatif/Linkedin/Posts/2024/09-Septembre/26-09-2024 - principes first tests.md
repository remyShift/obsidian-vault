Tu sais écrire des tests unitaires, mais est ce que tu sais écrire des bons tests unitaires ? 👇🏼


Un bon test unitaire c’est quoi ?

C’est un test qui respecte les principes FIRST.

FIRST c’est pour :

➡️ Flash Mcqueen (ou Fast pour les puristes) : ton test doit être rapide comme l’éclair.

Si on prends de la dette de temps de quelques millisecondes pour chaque tests il faudra multiplier ces millisecondes par le nombre de tests et le nombre de fois qu’on veut les lancer.

Arrivé à un moment on sera découragé de les lancer car on aura trop de tests qui prennent trop de temps.

Et qui dit tests lancées moins souvent dis avancer à l’aveugle. 

➡️ Isolé ton test tu dois. 

Il ne dois dépendre de rien. Dans une suite de tests ou seul son résultat doit être inchangé. 

De plus on doit pouvoir les lancer dans n’importe quel ordre sans que ça affecte les autres tests.

➡️ Ce qui permets d’avoir des tests Répétables.

Le test doit s’exécuter comme un système fermé sans dépendre de valeurs aléatoires.

On pourra ainsi le répéter à l’infini en ayant toujours le même résultat.

➡️ Self-verifying : le test n’est pas là que pour exécuter du code de production.

Il doit produire un résultat objectivable qui sera soit un échec soit une réussite.

De plus ce résultat doit être facilement identifiable.

➡️ Et pour finir le T est pour Timely !

Le test doit être écrit au moment opportun, et si on suit le TDD ça veut dire avant l’écriture du code de production.

Et aussi le test se doit d’être précis, il se doit de couvrir qu’un seul cas d’utilisation.

Pour écrire de bon tests unitaires respecter ces principes tu dois ! 🫵🏼
