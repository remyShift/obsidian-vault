# ADR-012: Analytics - two distinct event patterns (behavioral vs transactional)

## Context

The initial approach was to track all analytics events "as close to the click as possible". This caused `purchase` events to fire when the user clicked "Pay", before the backend confirmed the transaction. Failed orders were counted as sales in both Google Analytics and Meta. Additionally, Apple Pay does not expose cart items in the success page context, creating a second class of tracking problem specific to express checkout flows.

## Decision

Two distinct event patterns based on what the event represents:

**Pattern 1 - Behavioral events** (track as close to user action as possible):
- `view_item`, `begin_checkout`, `add_address`, `add_payment_details`, `search`, `skin_quiz_start`
- Client-side, fired on user action, PostHog only or PostHog + GA
- No backend confirmation required

**Pattern 2 - Transactional events** (track only when backend confirms):
- `purchase`, `add_to_cart` (by value), `subscribe`
- Must reflect backend reality. A failed transaction must never produce a `purchase` event.
- Long-term target: server-side tracking

## Options Considered

### Option A: All events client-side, close to user action (original approach)

- **Pro:** Simple, uniform pattern.
- **Con:** Transactional events fire even when the transaction fails. Data in GA and Meta is corrupted.

### Option B: Two patterns by event type (chosen)

- **Pro:** Behavioral events remain responsive (close to click). Transactional events are accurate (backend-confirmed). Data integrity in GA and Meta is restored.
- **Con:** Requires developers to make an explicit decision per event about which pattern applies.

## Rationale

Data integrity for transactional events directly affects marketing spend decisions. If failed purchases are counted as conversions, Meta and Google Ads optimise toward the wrong signal. The cost of a purchase event firing one second later (after backend confirmation) is zero. The cost of a fake conversion event is wasted ad budget.

## Consequences

**Positive**
- `purchase` events in GA and Meta now reflect real completed transactions.
- `logistic_order_id` (BigBlue) included in the purchase event payload for warehouse reconciliation.
- Clear mental model for developers: behavioral = fire on click, transactional = fire on backend confirmation.

**Negative / Accepted trade-offs**
- Apple Pay hotfix limitation: `purchase` event for Apple Pay flows lacks item-level data because the cart is not accessible in the success page context. Tracked with `logistic_order_id` + value only. Distinguishable via `payment_method` metadata.
- Local storage guard (`purchase_tracked: true`) is required to prevent duplicate events on page refresh.
- Refactoring transactional events to server-side (the long-term target) is a separate workstream not yet scheduled.
