# ADR-014: Categories - dedicated MongoDB collections replacing embedded translation field

## Context

Categories and subcategories were stored as a hardcoded `translation` field on the `shop_products` documents. This prevented clean multilingual support, made querying by category unreliable, and mixed commercial display grouping (what the user sees in the menu) with product-level categorization (used by the recommendation engine and analytics). There was also no way to attach metadata (image, translated name) to a category itself.

Two distinct categorization levels must be maintained:
- Product-level (DB): precise slugs (`hand_care`, `foot_care`, `moisturizer`, `serum`) for recommendation, routine builder, and analytics.
- Commercial-level (UX): grouping of subcategories for menu display (`hand_care` + `foot_care` -> "Hand & Foot Care"), used only for the storefront.

## Decision

Create dedicated `categories` and `subcategories` MongoDB collections. Product-level categorization uses the collection references. Commercial-level grouping is handled by a backend mapper (short-term), with a dedicated collection with translated names and images as the long-term target.

Migration plan: 3 PRs (schemas + seed script, backend API changes, frontend cleanup). An audit of all code locations reading `translation.categories` was required before PR 2.

## Options Considered

### Option A: Hardcoded translation field (status quo)

- **Pro:** Already in place.
- **Con:** No metadata possible on categories. Querying by category requires string matching on a nested field. No single source of truth for category names. Not extensible for multilingual support.

### Option B: Dedicated collections (chosen)

- **Pro:** Categories are first-class documents with their own ID, name (localized), image, and slug. Product-level and commercial-level categorization are explicitly separated. New automation (Make) looks up ObjectId by name from the collection.
- **Con:** Migration effort across backend, frontend, and Make automation. All Notion-sourced products must use category names that match exactly what is in the collection.

## Rationale

The `translation` field approach was always a short-term workaround. It made the product recommendation engine and analytics depend on string matching rather than stable IDs. The dedicated collections give each category a stable `_id` that can be used as a foreign key. The Make automation lookup by name is an acceptable constraint - category names are stable and rarely change.

## Consequences

**Positive**
- Categories have metadata: translated names, images, slugs.
- Product filtering queries use `_id` references instead of string matching.
- Clear separation between product-level categorization (analytics/recommendation) and commercial-level display.

**Negative / Accepted trade-offs**
- Make automation must look up category ObjectId by name on every product creation. Category names must match exactly.
- The commercial-level mapper (backend `if/else` grouping) is a short-term solution that still exists. The long-term dedicated collection for commercial groupings has not been built.
