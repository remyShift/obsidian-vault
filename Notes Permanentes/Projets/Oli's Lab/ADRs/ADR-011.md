# ADR-011: Analytics - custom client-side wrapper service instead of GTM

## Context

The product needs simultaneous event tracking across three platforms: PostHog (product analytics and heatmaps), Google Analytics (campaign optimization, purchase data), and Meta Pixel (Instagram conversion data). Each platform needs to receive the same event data. A dispatcher mechanism was needed to avoid firing three separate calls from every event handler in the code.

## Decision

Build a custom client-side `AnalyticsService` wrapper that dispatches events to all platforms from a single call site. No Google Tag Manager.

```ts
dispatchEvent('purchase', {
  value: 99.00,
  currency: 'EUR',
  items: [...],
  logistic_order_id: 'xxx'
})
// dispatches to PostHog + gtag (GA) + Meta Pixel simultaneously
```

## Options Considered

### Option A: Google Tag Manager

- **Pro:** No-code way to add/remove platforms. Non-dev team can configure events.
- **Con:** Requires manual value remapping (SKU, amount, items) for each platform inside the GTM UI. Adding a new event or changing a payload means coordinating a GTM config change + a code change. One extra layer of abstraction to debug. A PostHog -> GTM integration would need additional setup. Tag fires are harder to audit in code review.

### Option B: Custom wrapper service (chosen)

- **Pro:** Single source of truth for event payloads. Type-safe. Code-reviewable. Adding a new platform is one function call in the service. No external UI dependency for event configuration. Easier to test.
- **Con:** All event wiring is done in code - non-devs cannot add tracking without a PR.

## Rationale

For a small engineering team, the maintenance overhead of GTM (a separate system to configure correctly for every event change) outweighs the benefit. The custom wrapper is simpler to audit, test, and extend. Events must be asynchronous and must never block the UX - this constraint is enforced at the wrapper level.

## Consequences

**Positive**
- All analytics configuration lives in code, version-controlled, and reviewable.
- Asynchronous dispatching is enforced at the wrapper level.
- Adding or removing a platform is a single change in the service.
- Consistent event shape across all three platforms.

**Negative / Accepted trade-offs**
- Non-technical team members cannot add tracking without a developer.
- When the service is refactored for Next.js (planned between phase 0 and phase 1), both the CRA and Next.js streams will coexist temporarily.
