# ADR-003: CMS media - two separate Payload collections for raster and SVG

## Context

Payload CMS needs to store two fundamentally different types of visual assets: raster images (product photos, brand backgrounds, banners) and SVG logos. These have opposite requirements: raster images need format conversion (AVIF), resizing, focal point, and CDN URL rewriting; SVGs are vectorial, must not be resized, and have no focal point concept.

## Decision

Create two separate Payload collections: `Media` (raster images, AVIF only, thumbnail size, focal point enabled) and `LogoMedia` (SVG files only, no resize, no focal point). Both use the same S3 bucket with distinct prefixes (`media/` and `logo-media/`). Both have a `afterRead` hook that rewrites URLs to the CloudFront CDN.

## Options Considered

### Option A: Single collection with conditional branching

- **Pro:** Simpler config surface.
- **Con:** Conditional logic (if SVG, skip resize; if raster, apply focal point) makes the collection schema messy. Upload validation would need to reject SVG in one context and raster in another within the same form.

### Option B: Two separate collections (chosen)

- **Pro:** Clear separation of concerns. Each collection validates its own accepted formats. No conditional logic. UI is explicit about what each collection stores.
- **Con:** Slightly more config to write and maintain.

## Rationale

SVGs and raster images have enough diverging constraints (format, resize, focal point) that merging them into one collection creates more complexity than it solves. Two collections is the cleaner model and maps directly to how the content editors will use them.

## Consequences

**Positive**
- Upload validation is strict per collection (AVIF-only for Media, SVG-only for LogoMedia).
- No conditional logic in collection config.
- Relations in other collections (`brand.backgroundImage -> media`, `brand.logoImage -> logo-media`) are type-safe.

**Negative / Accepted trade-offs**
- Two configs to update if CDN URL pattern ever changes (mitigated: both share the same `afterRead` hook pattern).
