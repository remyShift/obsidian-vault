# ADR-013: Cart storage - database as single source of truth

## Context

In the CRA, the cart state is split across two storage mechanisms: some users have their cart in the database, others only in localStorage. The schemas diverged over time. This split caused concrete post-release bugs in March 2026: items remaining visible after purchase, cart displaying items no longer in localStorage, empty cart on page refresh, and adding one product replacing the entire cart. Additionally, Next.js App Router (SSR) cannot read localStorage on the server, making localStorage-based cart incompatible with the target architecture.

## Decision

The cart is stored exclusively in the database. localStorage is no longer a valid cart storage layer. A cart endpoint in the API manages all cart operations. This is a prerequisite for the Next.js checkout.

## Options Considered

### Option A: localStorage only

- **Pro:** No backend calls for cart operations, instant client-side updates.
- **Con:** Not accessible server-side (SSR incompatible). Lost on browser data clear. Cannot be shared across devices. Already proven to cause desync bugs.

### Option B: localStorage + DB sync (hybrid, current broken state)

- **Pro:** Offline-first feel.
- **Con:** Two sources of truth = desync bugs. Schema divergence over time. Hard to debug. Already causing user-visible bugs in production.

### Option C: Database only (chosen)

- **Pro:** Single source of truth. Works server-side (SSR compatible). Consistent across devices. No desync bugs. Stock validation can be done at persistence time.
- **Con:** Requires network call for every cart operation. Slightly higher latency for add-to-cart feel (mitigated with optimistic UI).

## Rationale

The hybrid approach already proved broken in production. SSR requires server-accessible state. The only viable path for the Next.js migration is DB-only storage. The latency concern is mitigated by optimistic UI updates on the client.

## Consequences

**Positive**
- Cart bugs caused by storage desync are eliminated.
- Cart accessible from any device (cross-device sessions).
- SSR can read cart state server-side for checkout rendering.
- Consistent schema for all users (no schema divergence).

**Negative / Accepted trade-offs**
- Cart operations require a network call. Optimistic UI is required to maintain perceived responsiveness.
- Users who had cart items only in localStorage lose them when their cart is migrated to DB-only. This is accepted as a one-time migration cost.
