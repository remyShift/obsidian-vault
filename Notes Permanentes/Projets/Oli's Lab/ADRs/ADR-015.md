# ADR-015: CMS translation - Payload plugin over custom solution

## Context

Payload CMS manages content in 4 locales (en, fr, it, es). A translator (Suze) will review all translations, but a first-pass automatic translation is needed so she does not start from scratch. Auto-generation of SEO metadata (title, description) per content item is also required. The question is whether to build a custom OpenAI integration or use existing Payload plugins.

## Decision

Use the existing Payload translation plugin for automated translation and `@payloadcms/plugin-seo` with `generateTitle`/`generateDescription` functions powered by OpenAI for SEO generation. No custom-built solution.

Workflow:
1. All content is created and finalized in English.
2. Editor clicks "translate" in the CMS admin.
3. Plugin translates all localized fields to French (and eventually other locales).
4. Suze reviews and corrects.
5. If the English source is updated after French translation, the French fields are retranslated (manual corrections are overwritten - accepted risk).

OpenAI model: cheap tier (mini-class model). Estimated cost: ~$10/month. An existing Oli's Lab OpenAI token is in the codebase (YAML files).

## Options Considered

### Option A: Custom OpenAI integration

- **Pro:** Full control over prompt engineering, tone of voice, field-level customization.
- **Con:** Code to write, test, and maintain. Content creation is sporadic (~monthly), so a custom solution would be over-engineered for the usage frequency.

### Option B: Payload plugin (chosen)

- **Pro:** Out-of-the-box functionality. No maintenance overhead. Supports Lexical rich text. `@payloadcms/plugin-seo` already exposes the hooks needed to inject OpenAI. Both plugins are installable independently.
- **Con:** The translation plugin repo is unmaintained for 2 years. Limited customization if specific tone-of-voice requirements emerge later.

## Rationale

Content creation frequency is low (monthly). Building and maintaining a custom solution for a feature used that rarely has no justifiable ROI. The plugin provides the functionality needed today. If the team needs more control in 6 months, a custom solution can be built then with the actual requirements in hand. The unmaintained state of the plugin repo is a known risk, accepted for an internal tool.

## Consequences

**Positive**
- Translation available in Payload admin UI out-of-the-box.
- SEO metadata generated automatically per document.
- Suze has a baseline to work from instead of translating everything from scratch.
- Both plugins are independent: SEO generation can be installed without the translation plugin.

**Negative / Accepted trade-offs**
- Updating the English source field after translation has been done overwrites Suze's manual corrections in French. Process mitigation: finalize English before triggering translation.
- The translation plugin repo has not been maintained for 2 years. Compatibility with future Payload versions is not guaranteed.
- Non-localized relation fields (e.g. `subcategory`) are not translated by the plugin - this is expected and acceptable.
