# ADR-006: API runtime - persistent Node.js server over AWS Lambda

## Context

The existing API runs on AWS Lambda (serverless functions). Lambda cold starts are problematic for user-facing latency, especially during traffic spikes when many simultaneous cold starts occur. The move to Next.js App Router (ADR-005) also favors a persistent server runtime for consistent response times and simpler deployment.

## Decision

API v2 targets a persistent Node.js server (not Lambda), deployed as part of the monorepo alongside the Next.js frontend. The server runs continuously, eliminating cold starts.

## Options Considered

### Option A: AWS Lambda (status quo)

- **Pro:** Pay-per-request pricing, auto-scaling, zero idle cost.
- **Con:** Cold starts degrade UX at the worst moment (traffic spikes). Stateless model adds complexity for connection pooling (MongoDB). Next.js App Router assumes a persistent server environment.

### Option B: Persistent Node.js server (chosen)

- **Pro:** No cold starts. Consistent response times. MongoDB connection pooling works correctly. Aligns naturally with Next.js App Router deployment model.
- **Con:** Always-on cost even at low traffic. Requires explicit scaling strategy for high-traffic events.

## Rationale

The cold start problem is a recurring source of user complaints and a known bottleneck. For an e-commerce app where checkout latency directly impacts conversion, the trade-off of always-on cost is worth consistent performance. The persistent server also simplifies connection management with MongoDB.

## Consequences

**Positive**
- Eliminated cold start latency.
- MongoDB connection pools are stable and reused.
- Simpler deployment model: one process to manage instead of isolated function deployments.

**Negative / Accepted trade-offs**
- Higher baseline infrastructure cost compared to Lambda idle pricing.
- Requires a scaling plan for traffic spikes (horizontal scaling or vertical upgrade).
