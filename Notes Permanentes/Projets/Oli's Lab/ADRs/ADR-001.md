# ADR-001: Payload CMS as content source of truth

## Context

Notion + Make was used as the CMS. It had no native CRUD interface, complex and fragile translation management, and a strong coupling with the Make automation that could silently overwrite fields on any trigger. As the product catalogue grew, the system became increasingly hard to maintain and impossible to type-check. There was also no way to model relationships between collections cleanly.

## Decision

Replace Notion + Make with Payload CMS as the single source of truth for all editable content: products, brands, categories, subcategories, articles, actives, concerns, skin types. The migration runs in parallel with the legacy CRA, using a bridge sync script (Payload -> MongoDB legacy) during the transition.

## Options Considered

### Option A: Notion + Make (status quo)

- **Pro:** Already in place, no migration effort.
- **Con:** No native CRUD, automation overwrites data unpredictably, no TypeScript types, hard to model relationships, complex translations.

### Option B: Payload CMS (chosen)

- **Pro:** Code-first, TypeScript types auto-generated from schema, self-hosted, full DB ownership, relations between collections, native drafts and versioning, multilingual support built-in.
- **Con:** Migration effort, new infra to set up and maintain.

### Option C: SaaS CMS (Sanity, Contentful)

- **Pro:** Managed hosting, no infra to run.
- **Con:** Subscription costs per member and per feature, vendor lock-in, no direct DB access, not adapted to a product that is primarily an e-commerce (not a content platform).

## Rationale

Payload gives full ownership of the schema and the data, generates TypeScript types automatically, and is free and open source. The SaaS alternatives were rejected because Oli's Lab is an e-commerce product first - what is being built is a scanner, a shop, an actives database - and paying per-seat for a hosted CMS did not make sense given that control and direct DB access matter.

## Consequences

**Positive**
- Schema is version-controlled code.
- Types are always in sync with the data shape.
- No risk of automation silently overwriting fields.
- Rich admin UI generated from the schema at no extra cost.
- Native multilingual support (4 locales: en, fr, it, es).

**Negative / Accepted trade-offs**
- Migration is a multi-month effort running in parallel with active development.
- A bridge sync script is required during transition to keep the legacy MongoDB in sync.
- The team must learn Payload conventions (hooks, access control, collections config).
