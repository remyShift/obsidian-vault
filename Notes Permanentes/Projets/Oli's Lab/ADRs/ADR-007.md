# ADR-007: Search - dedicated CRA page instead of Next.js modal

## Context

Search is a dependency of the Next.js navbar, which is already migrated. The CRA has the full search implementation (endpoints, logic, data shapes). Porting search to Next.js would require rewriting all data fetching against the Payload data shape, which is not yet stabilized (not all collections are on Payload). The existing search modal code in `expand.tsx` is tightly coupled to the checkout flow and is in poor shape.

## Decision

Build a dedicated search page in the CRA from scratch. Both navbars (CRA and Next.js) redirect to this CRA page when the search icon is clicked. The page is not a modal. Rémy leads the implementation. Kyle reviews PRs.

## Options Considered

### Option A: Port search modal to Next.js

- **Pro:** Keeps all new development on Next.js.
- **Con:** Requires rewriting all data fetching against the Payload shape, which is not yet finalized. Mixing two data sources (Payload and legacy MongoDB) on the same results page would create unpredictable behavior. Would take significantly longer.

### Option B: Dedicated CRA page from scratch (chosen)

- **Pro:** Reuses all existing endpoints and data shapes. No dependency on Payload readiness. Fastest path to a functional search experience. Clean codebase since it is built from scratch rather than patching `expand.tsx`.
- **Con:** Adds a page to the CRA that will eventually need to be migrated to Next.js when Payload is the full source of truth.

### Option C: Fix and extend the existing CRA modal

- **Pro:** No new page to route to.
- **Con:** `expand.tsx` is coupled to checkout logic and too messy to extend safely.

## Rationale

Path of least resistance given the current state of the Payload migration. Building in the CRA avoids mixing two data sources with different shapes. The existing modal code is too messy to extend, so starting fresh gives a cleaner result faster. The page can be migrated to Next.js in a future iteration when Payload covers all searchable entities.

## Consequences

**Positive**
- Delivers search quickly without blocking on Payload migration progress.
- Clean implementation starting from scratch.
- Works from both navbars via a simple redirect.

**Negative / Accepted trade-offs**
- Creates a CRA page that will need to be migrated to Next.js later.
- Search is not available as an inline experience (modal) - always a full-page redirect.
- Cannot use Payload-native search (Atlas Search, vector search) until the migration is complete.
