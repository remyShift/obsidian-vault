# ADR-016: API v2 backend pattern - load / execute / return

## Context

As the API v2 is being built with TypeScript, a consistent structure for service methods was needed. Without a clear pattern, service methods tend to mix database calls with business logic, making them hard to test and hard to reason about. Diego established this pattern during the code review of the first API v2 services.

## Decision

All service methods in API v2 follow a strict three-step structure:

```ts
// 1. load - DB call only, no business logic
const cart = await cartRepository.findById(cartId)

// 2. execute - pure business logic, no DB calls
const updatedCart = applyDiscount(cart, code)

// 3. return - return the result
return updatedCart
```

The controller does not validate inputs. Validation is handled by Joi middleware on the router. One Joi schema per endpoint when 2+ fields need validation; inline validation acceptable for a single field.

## Options Considered

### Option A: Mixed service methods (no separation)

- **Pro:** Less boilerplate, fewer files.
- **Con:** Business logic and DB calls are interleaved. Testing requires a MongoDB connection even for pure logic. Methods grow without clear boundaries.

### Option B: Load / execute / return (chosen)

- **Pro:** The `execute` step is pure (no DB calls), meaning it is unit-testable without MongoDB. Integration tests cover `load` (DB layer). The separation is enforced by convention and visible in the code.
- **Con:** Requires discipline. Methods that seem simple may feel over-structured initially.

## Rationale

The primary benefit is testability: pure functions in `execute` can be tested without a database. This is the only realistic path to meaningful unit test coverage on an API that currently has none. The secondary benefit is readability: any developer can look at a service method and immediately identify the data loading step from the business logic step.

## Consequences

**Positive**
- Business logic in `execute` is testable without MongoDB.
- Integration tests cover the `load` step (real DB required).
- Consistent structure across all services - less cognitive overhead when reading code.
- Controller responsibility is minimal: orchestrate calls, do not contain logic.

**Negative / Accepted trade-offs**
- Requires discipline to maintain the separation. A developer adding a second DB call inside `execute` breaks the pattern.
- Some simple operations (e.g. a one-line find-and-return) feel over-engineered with this structure. Convention is to apply it anyway for consistency.
