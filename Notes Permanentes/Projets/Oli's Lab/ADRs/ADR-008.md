# ADR-008: Search API - unified endpoint replacing 5 parallel calls

## Context

The current search implementation fires 5 separate HTTP requests from the client simultaneously (products, bundles, brands, articles, actives). The 5 endpoints have inconsistent naming conventions, no Joi validation, and are hard to extend. As the search page is built from scratch (ADR-007), this is the right moment to consolidate the backend.

The existing endpoints are not changed to preserve backwards compatibility with current CRA usage.

## Decision

Add a new unified endpoint `GET /search?q=&lang=` that returns all entity types in a single response. Client makes one request. The 5 existing endpoints remain untouched. `validateInput.js` (currently empty middleware placeholder) is implemented as the shared validation layer.

Response shape:
```json
{
  "products": [...],
  "bundles": [...],
  "brands": [...],
  "articles": [...],
  "actives": [...]
}
```

## Options Considered

### Option A: Unified endpoint (recommended)

- **Pro:** Single network round-trip. Centralized validation. Consistent query param naming. `validateInput.js` gets implemented and becomes reusable.
- **Con:** More backend work upfront. Response payload is larger (mitigated by limiting results per category).

### Option B: Fix the 5 existing endpoints (add validation, rename)

- **Pro:** Lower risk, incremental.
- **Con:** Does not solve the 5 parallel request problem. Renaming endpoints breaks existing usage unless aliases are kept.

### Option C: Search via Payload / Atlas Search

- **Pro:** Full-text search with relevance scoring.
- **Con:** Payload migration not complete. Atlas Search requires a separate index to configure and maintain. Not the right time.

## Rationale

The new search page is a clean slate. Building it against a unified endpoint costs the same as building it against 5 separate endpoints, but produces a better-structured, more maintainable result. The existing endpoints are not deleted, so there is no regression risk for the rest of the CRA.

## Consequences

**Positive**
- Search page makes one HTTP request instead of 5.
- Single place to add authentication, rate limiting, or caching on search in the future.
- `validateInput.js` finally has an implementation.

**Negative / Accepted trade-offs**
- Two backend search paths coexist temporarily (old 5 endpoints + new unified endpoint).
- Old endpoints are now dead code once the CRA modal is decommissioned.
