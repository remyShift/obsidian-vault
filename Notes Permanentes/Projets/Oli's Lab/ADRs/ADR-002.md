# ADR-002: CMS hosting - ECS Fargate for production

## Context

Payload CMS needs a hosting environment. During the exploratory phase, Lightsail was used as a quick starting point. As the CMS moved toward production readiness, a more robust and maintainable deployment target was needed. Diego set up the infrastructure.

## Decision

Use ECS Fargate (managed container service) for production CMS, with an EC2 t3.small instance for dev and exploration. All infrastructure in `eu-west-3` (Paris). Deployments triggered via GitHub Actions ("Deploy CMS" workflow) with a `target` parameter (`ecs` or `ec2`). Docker images pushed to ECR (`olis-cms` for prod, `olis-cms-dev` for dev).

Secrets are never in code or env files. They are stored in AWS SSM Parameter Store and injected at container startup.

## Options Considered

### Option A: Lightsail (abandoned)

- **Pro:** Very simple setup.
- **Con:** Limited scaling, no managed container support, used only as exploratory vehicle.

### Option B: EC2 (dev only)

- **Pro:** Simple, stoppable to reduce cost.
- **Con:** Manual scaling, unmanaged, not production-grade.

### Option C: ECS Fargate (chosen for prod)

- **Pro:** Managed containers, auto-scaling, ALB included, integrates natively with SSM for secrets, standard AWS production setup.
- **Con:** More complex than EC2 to configure initially.

## Rationale

ECS Fargate is the natural production target for a Dockerized Node.js app in AWS. It removes the operational overhead of managing EC2 instances while keeping costs proportional to usage. The SSM integration for secrets avoids any credentials in the codebase.

## Consequences

**Positive**
- No server management overhead in production.
- Secrets rotation without code changes (SSM only).
- Consistent Docker-based deploy from dev to prod.
- GitHub Actions deploy pipeline with environment targeting.

**Negative / Accepted trade-offs**
- ECS Fargate costs more than EC2 for always-on workloads.
- Slightly more complex initial setup and IAM configuration.
