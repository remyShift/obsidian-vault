# ADR-004: CMS sync - per-collection endpoints, not a generic script

## Context

During implementation of the Payload -> MongoDB legacy sync, a generic script was attempted that would handle all collections in a single loop. The script assumed a uniform mapping between Payload collections and MongoDB collections. This assumption broke on the first non-trivial case: subcategories do not exist as a MongoDB collection - they are keys inside a `categories` document object. Similar edge cases exist for other collections.

## Decision

One dedicated endpoint and one dedicated controller per collection for the Payload -> MongoDB sync. No generic script.

Architecture:
- CMS side: dynamic hook endpoint (`/api/sync/:collection`) that delegates to collection-specific logic.
- Server side: static routes per collection (`/internal/sync/brands`, `/internal/sync/categories`, etc.).
- Each collection has its own controller. No `if (collection === 'brands') { ... } else if (...)` in a shared controller.

## Options Considered

### Option A: Generic sync script

- **Pro:** Less code to write initially.
- **Con:** Breaks on the first edge case. Subcategories, products with draft status logic, and brands with tiles all have different shapes and MongoDB targets. A generic script becomes a maze of conditionals.

### Option B: Per-collection endpoints (chosen)

- **Pro:** Each collection is explicit and fully testable. Edge cases are handled locally, not mixed into shared logic. Adding a new collection does not risk breaking existing ones.
- **Con:** More files to create upfront.

## Rationale

The diversity of edge cases between collections makes a generic approach unsustainable. Diego's code review surfaced this when the generic script failed on subcategories. The cost of one controller per collection is lower than the maintenance cost of a generic controller with growing conditional branches.

## Consequences

**Positive**
- Each sync flow is independently readable and testable.
- A bug in brand sync cannot affect category sync.
- New collections can be added without touching existing controllers.

**Negative / Accepted trade-offs**
- More boilerplate per collection.
- A pattern change (e.g. changing how `syncMetadata` is updated) must be applied to each controller individually.
