# ADR-005: Frontend migration from CRA to Next.js App Router

## Context

The current Create React App has four structural problems. First, no SSR: all rendering is client-side, which hurts SEO and Time to First Contentful Paint on a catalogue that needs to be indexed. Second, AWS Lambda cold starts are slow under traffic spikes, degrading UX at the worst possible moment. Third, state management is scattered across Jotai atoms, local state, and localStorage with no clear boundaries. Fourth, 70% of users are on mobile but the CRA was built desktop-first. The platform is growing and the technical debt is compounding.

## Decision

Migrate the core user journey (product discovery, PDP, add to cart, checkout) from CRA to Next.js App Router with SSR. The CRA remains maintained in parallel by Patrick and Lucas during the transition. Kyle owns the Next.js Foundation. The migration follows a phased approach: Foundation (cart, checkout), PDP, PLPs, Homepage.

## Options Considered

### Option A: CRA (status quo)

- **Pro:** No migration effort.
- **Con:** No SSR, no SEO, Lambda cold starts, desktop-first CSS, growing state management debt.

### Option B: Next.js App Router (chosen)

- **Pro:** Native SSR/SSG, better image handling, mobile-first with Tailwind, monorepo-compatible, aligns with the persistent server decision (ADR-006). Components built stateless for SSR are also easier to port later.
- **Con:** Team split required, migration runs in parallel with active feature development for months.

### Option C: Remix

- Not evaluated. Next.js was the team's natural choice given existing React expertise.

## Rationale

SSR is the core requirement driving this migration - without it, the Payload CMS migration (which produces well-structured content) has no SEO payoff. Mobile-first is non-negotiable with 70% mobile users. Next.js App Router with Tailwind covers both. The phased approach (checkout first, then product pages) limits risk by starting with a flow that can run on a separate URL before the full migration.

## Consequences

**Positive**
- Server-side rendering for SEO and LCP improvement.
- Mobile-first by default with Tailwind standard breakpoints.
- Shared UI package `@olislab/ui` usable across CRA and Next.js during transition.
- Checkout v2 delivered as a first isolated Next.js feature (proof of concept in production).

**Negative / Accepted trade-offs**
- Two frontend codebases to maintain simultaneously for several months.
- Components must be built stateless (no hidden side effects) to work correctly with SSR - this constraint changes how some features are built.
- CRA bugs and Next.js new features must be prioritized separately.
